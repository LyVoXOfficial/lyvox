# Chat & Conversations Domain

## Overview
- Realtime диалоги продавец↔покупатель без перезагрузки страницы на Supabase Realtime (websocket `postgres_changes`).
- Разделение ответственности: SSR получает стартовые данные, подписка живёт только в клиентском компоненте (`use client`).
- Интеграции: профили, объявления (контекст сделки), модерация/жалобы, логи, уведомления.
- Связанные документы: [domains/profile.md](./profile.md), [domains/adverts.md](./adverts.md), [domains/moderation.md](./moderation.md), [requirements.md](../requirements.md), [API_REFERENCE.md](../API_REFERENCE.md), [Production master](../MASTER_PRODUCTION_TZ.md).

## Data Model (proposed)
- `public.conversations`:
  - `id uuid primary key default gen_random_uuid()`
  - `created_by uuid not null` → `public.profiles.id`
  - `advert_id uuid null` → `public.adverts.id` (если чат по объявлению)
  - `last_message_at timestamptz`, `created_at timestamptz default now()`, `updated_at timestamptz`
- `public.conversation_participants`:
  - `conversation_id uuid` → `public.conversations.id`
  - `user_id uuid` → `public.profiles.id`
  - `role text` (`owner`, `peer`), `muted boolean default false`, `last_read_at timestamptz`
  - PK (`conversation_id`, `user_id`)
- `public.messages`:
  - `id bigint generated by default as identity primary key`
  - `conversation_id uuid not null` → `public.conversations.id`
  - `author_id uuid not null` → `public.profiles.id`
  - `body text not null`
  - `created_at timestamptz default now()`, `updated_at timestamptz`
  - Индексы: (`conversation_id`, `created_at`), (`author_id`, `created_at`)
- Триггер `set_updated_at()` для `conversations`/`messages`.
- Связи с модерацией: сообщения можно помечать жалобами (см. ниже).

## RLS & Security
- `public.conversations`:
  - Select/Update: участники (`exists` в `conversation_participants` по `auth.uid()`) или админы (`public.is_admin()`).
  - Insert: любой аутентифицированный пользователь; на сервере проверять отсутствие дубликата 1:1-диалога.
- `public.conversation_participants`:
  - Select: участники/админы.
  - Insert/Delete: только сервис/создатель через server action (запрещаем самовольное исключение собеседника).
- `public.messages`:
  - Select: участники диалога.
  - Insert: `author_id = auth.uid()` и участник диалога.
  - Update/Delete: автор в небольшое окно времени или модератор (опционально, если поддерживаем редактирование/удаление).
- Все операции логируются в `public.logs` (действия `chat_send`, `chat_read`, `chat_mute`).

## API Surface (server actions / REST)
- `POST /api/chat/start` — создать/найти диалог (по `advert_id` и `peer_id`).
- `POST /api/chat/send` — вставка в `messages` c проверкой участия; возвращает нормализованный объект для оптимистичного UI.
- `POST /api/chat/read` — обновить `participants.last_read_at`.
- `GET /api/chat/history?conversationId=...&cursor=...` — пагинация сообщений, SSR для стартовой загрузки.
- Ограничения частоты: лимит на отправку (например, 20/мин/польз.) и суточная квота по IP.
- На сервере использовать `supabaseService()` только ПОСЛЕ проверки принадлежности пользователя к диалогу.

## Realtime Subscriptions
- Канал: `conversation:${conversation_id}` с `postgres_changes` на `public.messages` (INSERT/UPDATE при необходимости).
- Клиент: `use client` компонент (`ChatWindow.tsx`) или хук `useRealtimeMessages`:
  - `subscribe()` при монтировании, `unsubscribe()` при анмаунте.
  - Обработка `system` событий/ошибок, авто‑reconnect (экспоненциальная задержка).
  - Оптимистичные сообщения с последующей сверкой `id/created_at` из payload.
- SSR: получать стартовые N сообщений через `supabaseServer()` и куки.

## Notifications & Delivery
- События: новое сообщение, упоминание, подтверждение резерва/встречи (при связке с deals), массовые отключения канала.
- Каналы доставки: email; push (OneSignal/Firebase) — конфиг в инфраструктуре.
- Политика: не отправлять уведомления самому автору; дедупликация по 60 секунд; “quiet hours”.
- Хранение: лог отправок в `public.logs` с ключами `event`, `channel`, `status`.

## Retention & GDPR
- Предложение: хранить сообщения 6 месяцев после завершения сделки или удаления объявления; чаты без сделки — 6 месяцев неактивности.
- DSAR: экспорт истории в профиле по `conversation_id`.
- Удаление: soft‑delete записей с последующей очисткой кроном (Edge Function) и пересборкой индексов.

## Moderation & Abuse
- Жалобы на сообщения: связываем с `public.reports` через `target_type='message'`, `target_id=messages.id`.
- Авто‑mute: при X подтверждённых жалобах по участнику в рамках N дней — пометка `muted=true` в `participants`.
- В UI — кнопка “Пожаловаться” на сообщении, попадание в очередь модерации.
