# Internationalization & Locales (RFC)

## Overview
- Поддержка нескольких стран и языков (локали, валюты, форматы телефонов).
- Требуется для масштабирования и роста маркетплейса.

## Data Model (proposed)
- `public.locales` (`code`, `name`, `currency`, `date_format`, `phone_region`)
- `public.user_locales` (`user_id`, `locale_code`, `created_at`)
- Расширения в `adverts` для локализованных атрибутов (при необходимости) — или хранение в нейтральном формате с форматированием на фронте.

## App Architecture
- Next.js App Router без префиксов в URL; выбор языка хранится в cookie `locale`.
- Middleware `apps/web/middleware.ts` выставляет `locale` по query `?lang=` либо `Accept-Language` (персистит в cookie).
- Словари JSON: `apps/web/src/i18n/locales/{en,fr,nl,ru}.json`.
- Провайдер: `apps/web/src/i18n/index.tsx` (контекст `useI18n` с `t(key)`), серверные хелперы `apps/web/src/i18n/server.ts`.
- Встраивание: в `app/layout.tsx` поднимается `I18nProvider`, `html lang` = активная локаль, meta берутся из словаря.
- Форматирование дат/валют: стандартные Intl API, локаль берётся из контекста (добавить утилиты по мере необходимости).

## Supported Locales
- EN, NL, FR, RU, DE (fallback chain for missing keys: `de → en → ru`).

## Data normalization workflow (RU/EN mixed translation)
- Источник: исторические данные могли содержать русские названия/описания в `name_en`/`summary`.
- Скрипт `scripts/vehicle_i18n_normalize.mjs`:
  - Детектирует кириллицу и смешанные записи.
  - Переводит ru→en через OpenAI (переменная `OPENAI_MODEL`, по умолчанию `gpt-4o-mini`; допускается `gpt-5` при наличии).
  - Обновляет базовые `*_en`, а исходный русский текст складывает в `vehicle_*_i18n(locale='ru')`.
  - Логи правок — `logs/vehicle_i18n_autotranslate.csv`.
- Сохранность имён собственных (марка/модель) обеспечивается инструкцией для модели (не транслитерировать названия: Passat остаётся Passat).

## Deep Content Localization (inspection_tips, pros/cons, issues)
- Где хранится: `public.vehicle_generation_i18n` дополняется колонками:
  - `pros text[]`, `cons text[]`, `inspection_tips text[]`, `common_issues text[]`, а также `summary text` (уже была).
- Как пополняется: скрипт `scripts/vehicle_i18n_expand.mjs`
  - Источник: базовые поля поколения/модели (`vehicle_generations.summary`, `vehicle_insights.pros/cons/inspection_tips/common_issues_by_engine`).
  - Определение языка: RU vs EN по наличию кириллицы.
  - Генерация: для каждого поля создаются варианты для `en, ru, nl, fr, de` с сохранением тех. токенов (BMW/TDI/DSG/DPF и др.).
  - Запись: upsert c merge-логикой — не затирает существующие переводы; заполняет только пропуски.
  - DRY_RUN: поддерживается; аудит — `logs/vehicle_i18n_expand.csv` (generation_id, field_name, locale_written, original_snippet, translated_snippet).
- Fallback порядок при чтении: `запрошенная локаль → en → ru`.
- Зачем это важно: богатые описания повышают SEO (rich snippets) и релевантность ответов AI‑ассистентов в сценариях вроде “найди дизель до 5000 евро с автоматом”.
