# Сравнение blueprints: совет Claude vs совет Codex

Дата: 2026-07-03. Сравниваются два независимых экспертных совета по одной задаче («идеальный план структуры LyVoX»):
- **Claude** — [SITE_BLUEPRINT.md](SITE_BLUEPRINT.md): 6 ролей, 2 раунда (позиции + 54 перекрёстных возражения), opus-синтез, привязка к живому коду/данным.
- **Codex** — [FINAL_BLUEPRINT.md](../expert-councils/codex-16-agent-site-blueprint-2026-07-03/FINAL_BLUEPRINT.md): 16 агентов gpt-5.5 xhigh, раунды+голосование, отдельная проверка регуляторных источников; device-QA намеренно не выполнялся.

## 2. Что сильнее у Claude

1. **Привязка к реальному коду и данным.** Указаны конкретные файлы и строки (sitemap только level=1; blanket-lazy в ad-card; signMediaUrls ломает кэш на каждом ISR), пороги от фактических 960 объявлений, связка с GTM/бизнес-планом (пороги 1000 объявлений / 50 контактов/нед). Codex явно кодовую базу не примерял.
2. **Seed-гигиена как правовой риск №1** — у Codex тема отсутствует полностью, при том что каталог ~90% seed: seed-отзывы = per-se запрещённые фейковые отзывы (CEL VI.100 23°ter), seed в медианах/топах/JSON-LD, план вывода волнами. Это, вероятно, самая ценная находка из обоих документов.
3. **URL-локали.** Cookie-локаль = Google индексирует один язык → главный дифференциатор («5 языков») в органике не существует. У Codex — только «no untranslated duplicate pages», сам архитектурный дефект не замечен.
4. **Бельгийская правовая конкретика** в каждом приёме: CEL VI.97/VI.100, CRD Art.22 (предвыбранные допы ничтожны), Omnibus Art.7(4a), UCPD Annex I, DSA Art.25/26/27/30. У Codex — правильные принципы с общей формулировкой «нужен юрист».
5. **Таблица законных приёмов роста** (10 приёмов с обоснованием легальности каждого) — прямой ответ на запрос про «трюки по закону»; у Codex преобладают запреты, наступательных приёмов мало.
6. **Уникальные тактики, которых у Codex нет:** sold-страницы как trust-актив (200→410+анонимизация), share-loop после публикации, индексируемый лендинг /sell, харвест zero-result запросов, bfcache instant-back, конвейер изображений ДО редизайна, план по /discover (ложный empty-state, порог 3000).
7. **Волны внедрения + нерешённые трейд-оффы** для решения основателя — план управляем, а не только нормативен.

## 3. Что сильнее у Codex

1. **Контракт карточки** — исчерпывающий список содержимого карточки на home/search (фото/цена/заголовок/локация/свежесть/статус продавца/1 бейдж/save-report) как компонентное правило. У Claude есть анти-stacking, но не полный контракт. **→ принять.**
2. **Визуальная система** — у Claude этот слой почти отсутствует: ступени радиусов (20 только shells / 12–13 карточки / 8–9 контролы / pill только чипы) — точная критика текущего «20px везде» плюшевого вида; amber = caution и платное раскрытие; цены — Onest tabular (не моно), Geist Mono только для ID/KBO/таймстампов; тест длинных нидерландских/немецких композитов. **→ принять.**
3. **«Why shown» на каждой промо-карточке** — конкретнее, чем общая страница прозрачности. **→ принять** (плашка + tap → почему показано).
4. **Thin-inventory recovery как секция главной** (расширить радиус/категорию, сохранить поиск, все свежие) — у Claude это есть в /search и /c/*, но не на главной. **→ принять.**
5. **Чек-листы приёмки** (a11y: WCAG 2.2 AA + EAA, 400% reflow, no hover-only; i18n QA-список) — формат release-gate удобнее эссеистики. **→ принять как чек-листы к волнам.**
6. **Location в hero-поиске** (query+category+location) — для локального маркетплейса разумно. **→ принять как необязательный чип** (не третье обязательное поле — трение).
7. Позиционирующие формулировки на языке покупателя (2 варианта слогана) — полезная альтернатива.

## 4. Противоречия и их разрешение (арбитраж дирижёра)

| Вопрос | Claude | Codex | Решение |
|---|---|---|---|
| Отрицательный letter-spacing | Есть на display (−0.02em на h1/h2, уже в проде) | «Никогда» | **Компромисс:** −tracking только на display ≥28px (типографская норма для крупных гротесков), 0 на body/UI — правило Codex мотивировано читаемостью длинных NL/DE-композитов в мелком тексте |
| Блок «Как защищаем» | Позиция 4 (после товара и категорий) | Внизу, у футера | **Claude, с оговоркой:** trust-first бренд + тревожный покупатель оправдывают середину страницы; блок компактный (3 строки-механизма). Если A/B покажет вред конверсии — опустить |
| TopSellers | Трансформировать с порогом (≥4 продавца ≥3 отзывов, иначе null) | Вырезать | **Фактически одно и то же:** порог сейчас недостижим → секции нет; правило Claude оставляет ей законный путь вернуться |
| «Бесплатные» | Чип-фильтр над фидом | «All recent» ниже качественного грида | **Claude** (дешевле: ноль дополнительных секций/картинок), идея Codex «recent» покрывается сортировкой фида по свежести |
| Буст-слот на главной | Позиция 3–4 фида, макс 1, с плашкой | Не в первом proof-гриде (позже допустим с гейтами) | **Совместимо:** первый грид органический у обоих; слот Claude живёт в фиде ниже — принимается с правилами Codex (why-shown, те же trust-гейты, кап) |

## 5. Итог

**Качество:** документы удивительно согласованы по ядру (≈80% вердиктов совпали независимо — можно считать их валидированными дважды). Claude-версия выигрывает в **приземлённости** (код, данные, бельгийское право, seed-риск, локали, наступательные тактики, план волн) — она пригодна как рабочий план внедрения. Codex-версия выигрывает в **нормативной дисциплине** (контракт карточки, визуальная система, release-чек-листы) — она пригодна как система правил приёмки.

**Действие:** SITE_BLUEPRINT.md остаётся мастер-планом; из Codex в него принимаются 7 пунктов раздела 3 и арбитражи раздела 4 (letter-spacing-компромисс, why-shown, location-чип, thin-recovery секция, контракт карточки, ступени радиусов, чек-листы приёмки). Противоречий, требующих решения основателя, не осталось — обе «нерешённые» зоны (§ трейд-оффы SITE_BLUEPRINT) не затронуты Codex-документом.
