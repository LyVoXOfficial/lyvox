# PRD: In-chat оплата Bancontact (betaalverzoek)

> **Статус кода:** ⛔ НЕ НАЧАТО (флоу оплаты покупателем отсутствует)
> **Категория:** Не хватает (критическое)
> **Приоритет:** P0
> **Зависит от:** [[10-escrow-safe-deal]] (оплата идёт в холд, не напрямую продавцу), [[35-chat-antifraud]] (запрос оплаты — только нативный, не ссылкой)
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Bancontact — де-факто стандарт оплаты в Бельгии (≈30% предпочтений). 2dehands использует in-chat «betaalverzoek» как основной safe-path. У LyVoX оплаты покупателем нет вообще, а внешние payment-ссылки в чате — главный фишинг-вектор.
- **Исход.** Продавец/покупатель инициируют запрос оплаты **внутри чата**, оплата идёт в **escrow-холд** через Bancontact/карты (dynamic methods). Внешние payment-ссылки — детектируются и предупреждаются ([[35-chat-antifraud]]).
- **KPI:** доля оплат через нативный запрос; доля сделок с Bancontact; падение off-platform payment-попыток.

## 2. Контекст и текущее состояние (As-is)
- Stripe только для бустов; нет «запроса оплаты» в чате; чат-антифрод (`scrubContacts`) уже маскирует ссылки/телефоны — на это опираемся.

## 3. Пользователи и сценарии
- **Продавец:** «Договорились в чате — отправляю запрос оплаты на сумму X».
- **Покупатель:** «Вижу нативную карточку оплаты в чате, плачу Bancontact, деньги в холде».
- Happy-path: чат → «Запросить оплату» → карточка запроса (сумма, товар, безопасная сделка) → покупатель платит → статус «оплачено (в холде)» → сделка [[10-escrow-safe-deal]].

## 4. UX и поведение (детально)
- **Кнопка «Запросить оплату»** в чате (для продавца) → форма суммы/привязка к объявлению → в ленте чата появляется **нативная карточка оплаты** (не ссылка).
- **Оплата:** checkout с dynamic payment methods (Bancontact обязателен, карты, кошельки) → возврат в чат со статусом.
- **Анти-фишинг:** если кто-то шлёт внешнюю payment-ссылку — инлайн-предупреждение «LyVoX никогда не просит платить по ссылке вне приложения».
- **Состояния карточки:** запрошено → оплачено (в холде) → отменено/истекло.
- **Десктоп/мобайл:** одинаковая нативная карточка.

## 5. Логика и правила
- Запрос оплаты **всегда создаёт escrow-deal** (нет «прямой» оплаты продавцу в обход холда).
- Срок жизни запроса (истечение), отмена, повторная отправка.
- Идемпотентность: один запрос = один payment intent; статус — от вебхука.
- Лимиты: rate-limit на создание запросов (анти-абуз), потолок суммы на старте.

## 6. Данные (Data model)
- `payment_requests { id, conversation_id, deal_id, advert_id, seller_id, buyer_id, amount, currency, status, expires_at, created_at }` (или как событие сделки).
- Связь с `deal_payments` ([[10-escrow-safe-deal]]). RLS: участники чата/сделки.

## 7. API / интеграции
- `POST /api/chat/payment-request` (создать), `POST /api/deals/[id]/pay` (checkout, переиспользование escrow), `POST /api/webhooks/stripe` (статус).
- Stripe dynamic payment methods (Bancontact включён в Dashboard); не форсить `payment_method_types`.

## 8. Безопасность, приватность, комплаенс
- Только нативные запросы оплаты; внешние ссылки — фрод-сигнал ([[35-chat-antifraud]]).
- Все деньги — через escrow-комплаенс ([[10-escrow-safe-deal]] §8: KYC/AML/NBB-вопрос).
- Аудит создания/оплаты запросов.

## 9. Аналитика
- `payment_request_created/paid/expired/cancelled`; доля Bancontact; off-platform link attempts.

## 10. Доступность и i18n
- Карточка оплаты с ARIA/статусами; i18n `payment.*` (NL/FR/EN), локализованные суммы/валюта.

## 11. Тестирование
- API: создание/оплата/истечение + идемпотентность вебхука. e2e (test mode): запрос → Bancontact оплата → холд. Негатив: внешняя ссылка → предупреждение; повтор вебхука.

## 12. План внедрения (Rollout)
- Вместе с escrow (NOW-2); Bancontact в checkout — обязателен. Усилие: M.

## 13. Открытые вопросы и решения
- Кто инициирует запрос (продавец/обе стороны). Срок истечения. Потолок суммы на старте.

## 14. Критерии готовности (DoD)
- [ ] Нативный запрос оплаты в чате создаёт escrow-холд (не прямую оплату).
- [ ] Bancontact доступен в checkout; статус от вебхука идемпотентно.
- [ ] Внешние payment-ссылки помечаются предупреждением.
- [ ] Тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

> ⛔ Блокер ревью: де-факто заблокирован у LEGAL+SEC — неотделим от escrow-гейта PRD 10 (PSD2/AML F3 + идемпотентность F1 + authorization денег F2). Сам UX-флоу ок; sign-off только после закрытия PSD2-гейта.

### LEGAL
- **PSD2/AML (§8):** наследует гейт PRD 10 (F3); строить нельзя до письменного заключения. Зафиксировать «SCA обеспечивается PSP; LyVoX не обходит SCA» (PSD2 RTS — Bancontact/карты требуют SCA).
- **Pre-contractual / total price (§4):** до оплаты на нативной карточке показать total price вкл. fees, identity получателя средств (escrow/Stripe, НЕ продавец напрямую) и что это safe-deal; запрет формулировок «прямая оплата продавцу» (иначе misleading omission, UCPD).
- **Chargeback ↔ escrow (§5):** карточные платежи дают chargeback-права через issuer независимо от платформы; не выплачивать продавцу, пока открыто chargeback-окно по высокорисковым (анти-двойной-убыток).
- **Retention:** оплаченные `payment_requests` = 7 лет (BE bookkeeping); см. F4.

### DEV
- **Deal-first модель (зафиксировать однозначно):** запрос оплаты ВСЕГДА создаёт `deal` в `pending_payment` + строку `payment_requests` со ссылкой `deal_id` (убрать «или как событие сделки» — иначе двойная истина статуса).
- **F1 — идемпотентность (S0):** общий механизм с PRD 10 (`webhook_events` по Stripe `event.id`); `idempotency_key` на создание запроса + защита от двойного клика «Запросить оплату».
- **Инициатор (закрыть §13):** рекомендация — продавец создаёт, покупатель платит (проще для anti-fraud и UX-карточки).
- **Истечение:** `expires_at` + cron → `expired`; связанную сделку `pending_payment` → cancel.
- **Состояния (S4):** `requested → paid | cancelled | expired`. RLS через `is_conversation_participant()`.
- **Эндпойнты:** `POST /api/chat/payment-request` (rate-limit как `chat/send`; создаёт deal+request), `POST /api/deals/[id]/pay` (переиспользуется из PRD 10), вебхук — общий stripe-connect (S0).
- Dynamic payment methods — не задавать `payment_method_types` (Bancontact в Dashboard) — оставить как есть.

### SEC
- **Authorization суммы/получателя (Critical, S-7):** server-side — создатель payment_request == owner объявления (или явное правило инициатора); сумма ≤ потолок; buyer/seller привязаны к участникам conversation (verified server-side, не из клиента).
- **Replay/идемпотентность (Critical, S-3):** общий механизм с PRD 10 (processed `event.id`), иначе двойная оплата/задвоение холда.
- **Native-карточка только из server-entity (Med):** карточка оплаты рендерится строго из `payment_requests` (typed event/server-controlled message kind), не из произвольного текста сообщения — иначе спуфинг «карточки-самозванца».
- **Фишинг внешних ссылок:** `scrubContacts` тривиально обходим (слитные цифры, unicode, дробление) — честно зафиксировать: реальный контроль = escrow, scrub = deterrent/сигнал; усилить (NFKC, BE-номер `^0\d{9}$`, агрегация по conversation) — см. PRD 35.
- **Предусловия:** S-1 (серверный IP, F8) для rate-limit `POST /api/chat/payment-request`; S-2 (CSP enforced + nonce).

**Вердикт:** ⛔ — заблокирован escrow-гейтом PRD 10 (F1–F3); правки: deal-first модель, total-price/получатель disclosure, SCA, chargeback↔escrow, authorization суммы/получателя, native-card только из server-entity.
