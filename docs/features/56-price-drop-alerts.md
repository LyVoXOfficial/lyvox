# PRD: Price-drop алерты на избранном

> **Статус кода:** ⛔ НЕ НАЧАТО (есть избранное и saved-search алерты — переиспользуем)
> **Категория:** Бонус
> **Приоритет:** P2
> **Зависит от:** [[34-discovery-extras]] (избранное), [[36-notifications]]
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Покупатель ждёт снижения цены, но не возвращается → теряем сделку.
- **Исход.** При снижении цены товара из избранного — алерт (in-app/email/пуш). Возвращает пользователя в момент готовности купить.
- **KPI:** CTR price-drop алертов; возвраты по алерту; конверсия в контакт/сделку.

## 2. Контекст и текущее состояние (As-is)
- Есть `favorites`, нотификации и cron saved-search-alerts — инфраструктура переиспользуется; отслеживания изменения цены нет.

## 3. Пользователи и сценарии
- «Добавил в избранное → цена упала → получил алерт → вернулся».

## 4. UX и поведение (детально)
- Авто-включение для избранного (с тоглом в preferences).
- Алерт: «Цена снижена: было X, стало Y» + ссылка на товар.
- Состояния: отслеживается/сработал/товар снят.

## 5. Логика и правила
- Триггер при изменении `price` объявления в избранном (порог изменения, напр. ≥1% или ≥1€).
- Дедуп/частота/quiet hours ([[36-notifications]]); отписка.
- Краевые: повышение цены (не алертим), удалённый/проданный товар.

## 6. Данные (Data model)
- История цены: либо `advert_price_history { advert_id, price, changed_at }`, либо детект на апдейте цены. RLS публичное чтение цены; алерты owner-only.

## 7. API / интеграции
- Хук на изменение цены объявления → очередь алертов; нотификации.

## 8. Безопасность, приватность, комплаенс
- Opt-out; consent на канал; GDPR.

## 9. Аналитика
- Срабатывания, CTR, возвраты, конверсия.

## 10. Доступность и i18n
- i18n `price_drop.*` (NL/FR/EN); локализованные суммы.

## 11. Тестирование
- Unit: детект снижения/порог. e2e: снизить цену → алерт.

## 12. План внедрения (Rollout)
- **LATER:** дёшево поверх избранного+нотификаций. Усилие: S.

## 13. Открытые вопросы и решения
- Порог изменения. Дефолт on/off. Хранить ли полную историю цен.

## 14. Критерии готовности (DoD)
- [ ] Снижение цены избранного → алерт с дедупом/отпиской.
- [ ] i18n; тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

Ревью: критичных замечаний именно по этому PRD нет; применить сквозные F-требования где релевантно. Релевантные: алерты наследуют **матрицу каналов и дедуп из PRD 36** (`notification_deliveries` с `dedup_key = price_drop:advert_id:user_id:channel`, quiet hours по `profiles.timezone`) и UX-дефолты алертов из PRD 34 (max 1/день, пауза, отписка в 1 тап — анти-усталость); KPI CTR/возвраты пишутся в единый sink событий (F6, `saved_search_alert_clicked`-аналог); `advert_price_history` — opt-out/consent на канал и GDPR-retention (F4). Краевой кейс «товар снят/продан» — пометка «снято с продажи» + похожие (согласовать с UX краевых состояний PRD 34).

**Вердикт:** 🔄 — после наследования дедупа/quiet-hours из 36 и UX-дефолтов из 34 (+ sink событий F6) → ✅; самостоятельных блокеров нет.
