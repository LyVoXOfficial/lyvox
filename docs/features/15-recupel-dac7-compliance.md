## 1. Зачем это (Problem & Outcome)
- **Проблема.** С 29.03.2025 оператор онлайн-маркетплейса обязан информировать продавцов электроники об их EPR-обязанностях (Recupel/WEEE) и **проверять** их соответствие (или брать обязанности на себя). DAC7 требует сбора налоговых данных продавцов. Без гейтов — регуляторный риск с первого дня категории «электроника» + бизнес-продавцов.
- **Исход.** Продавец электроники-профессионал не публикует regulated-листинг без Recupel-данных; данные DAC7 собираются с первого дня (отчёт — годовой, отложен).
- **KPI:** 100% профессиональных электро-листингов с проверкой Recupel; полнота DAC7-данных у транзакционных продавцов; ноль нарушений по аудиту.

## 2. Контекст и текущее состояние (As-is)

## 3. Пользователи и сценарии
- **Профессиональный продавец электроники:** «Указываю Recupel-членство/EPR, иначе не могу опубликовать».
- **Любой транзакционный продавец:** «Предоставляю налоговые данные (DAC7) при достижении порога».
- **Админ/Compliance:** «Вижу статус Recupel и полноту DAC7; ставлю гейты».

## 4. UX и поведение (детально)
- **Recupel-гейт (публикация электроники профи):** поле членства/EPR + информирование об обязанностях; без валидного статуса — блок публикации regulated-листинга.
- **DAC7-сбор:** при первой транзакции/достижении порога — форма данных (идентификация, TIN/НДС, адрес), переиспользуя itsme-captured identity ([[30-identity-account]]).
- **Состояния:** не предоставлено / на проверке / подтверждено / истекло.
- **Информирование:** письменно уведомить продавцов об их EPR-обязанностях (требование оператора маркетплейса).

## 5. Логика и правила
- **Recupel:** гейт на публикацию электроники для профи-продавцов; периодическая перепроверка (expiry).
- **DAC7:** собирать сейчас, **отчёт — годовой** (первый — 31 января следующего года; pre-launch с ~0 продавцов = нет day-one отчётной нагрузки). Пороговая логика (число сделок/сумма).
- Краевые: продавец «частник» продаёт электронику разово (C2C — не EPR-обязан) vs профи (обязан) — связать с [[14-consumer-trader-rights]].

## 6. Данные (Data model)
- `seller_compliance_status { user_id, category, type ('recupel'|'dac7'|...), status, reason, expires_at }`
- `compliance_documents { id, owner_id, category, type, storage_path, status, expiry }`
- DAC7-поля: TIN/VAT, юр-данные (приватно, под RLS owner/admin). Retention/экспорт (GDPR), безопасное хранение чувствительного.

## 7. API / интеграции
- Гейт в `/api/adverts` (публикация электроники профи) — проверка `seller_compliance_status`.
- `POST /api/compliance/recupel`, `POST /api/compliance/dac7` (сбор/обновление).
- (Опц.) сверка Recupel-реестра; экспорт DAC7-отчёта (годовой, позже).

## 8. Безопасность, приватность, комплаенс
- **Recupel/WEEE (BE, 29.03.2025), DAC7 (EU)** — основной предмет; **GPSR** — смежно ([[38-moderation-reports]] обрабатывает нотисы об опасных товарах за 3 дня).
- Чувствительные налоговые данные — шифрование/минимизация, доступ только owner/admin, аудит.

## 9. Аналитика
- Доля электро-листингов с Recupel; полнота DAC7; число заблокированных публикаций по гейту.

## 10. Доступность и i18n
- Понятные объяснения обязанностей; i18n `compliance.*` (NL/FR/EN, юр-выверка).

## 11. Тестирование
- Unit: пороговая логика DAC7, expiry Recupel. API: гейт публикации без Recupel → блок. e2e: профи публикует электронику только после Recupel.

## 12. План внедрения (Rollout)
- **NOW-1:** DAC7-сбор (переиспользуя identity); Recupel-гейт при активации электроники для профи. Отчёт DAC7 — отложен. Усилие: M.

## 13. Открытые вопросы и решения
- Брать ли EPR-обязанности на себя vs только проверять продавцов. Источник верификации Recupel-членства. Точные пороги DAC7.

## 14. Критерии готовности (DoD)
- [ ] Профи-электро-листинг блокируется без валидного Recupel-статуса; продавцы письменно информированы.
- [ ] DAC7-данные собираются и хранятся безопасно (RLS/шифрование), готовы к годовому отчёту.
- [ ] Аудит + retention (GDPR).
- [ ] Тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

### LEGAL
- **EPR/WEEE — решить §13 (deemed-producer риск):** зафиксировать позицию по обязанности для (a) BE-профи (на них; LyVoX проверяет + информирует) и (b) **иностранных/незарегистрированных продавцов** — по WEEE-Dir Art.5 + KB 26.06.2024 / Recupel-regelgeving оператор маркетплейса, допускающий non-EU/non-registered EEE на BE-рынок, может сам стать «deemed producer» (регистрация в реестре производителей / Recupel-членство). Оценить с counsel; возможно блокировать non-registered foreign EEE-продавцов. Источник верификации членства — Recupel-реестр/API.
- **Recupel-взнос (visible fee):** для new/refurb EEE (трейдер) отображать administratieve bijdrage / contribution Recupel покупателю на товаре; для б/у обычно нет.
- **C2C информирование:** generic WEEE-инфо (правильная утилизация) для ВСЕХ электро-листингов, не только профи.
- **DAC7 — точные параметры (закрыть §13):**
  - Поля Reportable Sellers: физлица — имя, основной адрес, **TIN + страна выдачи**, дата рождения; юрлица — legal name, адрес, TIN, **KBO**, VAT; + financial account identifier (счёт получения средств).
  - **Due diligence/верификация** через гос-реестры (не просто сбор).
  - **Порог excluded sellers:** <30 сделок **И** <€2000 за год по продаже товаров → не reportable (т.е. >30 транзакций **ИЛИ** ≥€2000 → reportable). Зафиксировать именно эти числа.
  - **Уведомление продавца** о передаче данных в налоговую (DAC7 + GDPR-transparency Art.13/14).
  - **Enforcement:** после 2 напоминаний без данных → закрытие аккаунта / удержание выплат.
  - **Отчёт** до **31 января** следующего года в FOD Financiën / SPF Finances, XML по EU-схеме (сбор + верификация — с первой транзакции).
- **GDPR налоговых данных:** основание = **Art.6(1)(c)** (правовая обязанность DAC7), НЕ consent; TIN/VAT/DOB — BE national-ID особый режим; retention по налоговому сроку (см. F4). RoPA/DPA-фундамент F4.
- **GPSR (Reg. 2023/988, с 13.12.2024) — усилить §8:** Art.22 обязанности online marketplace — single point of contact (для authorities и потребителей), notice-and-action по опасным товарам, кооперация с Safety Gate, **takedown опасных товаров «without undue delay»** (НЕ фикс 3 дня для опасных — исправить формулировку в §8); traceability (производитель/импортёр/EU responsible person, Reg. 2019/1020) для new/trader-товаров. Кросс-ссылка на PRD 38/12.
- **DoD добавить:** EPR-позиция зафиксирована юристом; DAC7 due-diligence + seller-notice + enforcement реализованы; GPSR point-of-contact + notice-and-action активны.

**Вердикт:** 🔄 — EPR-обязанность не решена (deemed-producer риск), DAC7 due-diligence/notice/enforcement/пороги не специфицированы, GPSR-обязанности маркетплейса недооценены. Sign-off после фиксации EPR-позиции и DAC7-параметров.
