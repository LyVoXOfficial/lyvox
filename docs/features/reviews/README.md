## 2. Жёсткие блокеры (⛔) — без них область не закрывается
- **PRD 10 Escrow** — ⛔ у LEGAL, DEV, SEC. Корень: F3 (PSD2/AML) + F1/F2 (идемпотентность/авторизация). Тянет за собой 11 и 13.
- **PRD 62 Карточки/KB** — ⛔ у DEV, DATA, SEO. Баг 1996 ЖИВ в коде (`determineGeneration()` `.find()` при overlap); нет structured data; вкладки нереализуемы текущим рендерером. → F7, F12, F13.
- **PRD 37 Trust** — ⛔ у DATA. Формулы нет вовсе. → F14.
- **PRD 31 Создание объявления** — ⛔ у UX. Консервирует 8 шагов + барьер логин→верификация до формы (тормоз supply). Нужен черновик-до-верификации + адаптивные шаги.
- **PRD 33 Поиск / 32 Каталоги** — ⛔ у SEO. `/search` индексируется и `"use client"` (нельзя metadata); SEO категорий `/c/*` отсутствует. → F12.
- **PRD 13 In-chat оплата** — ⛔ у LEGAL+SEC (неотделим от escrow F1–F3).

## 4. Важные правки к самим PRD (фактические неточности, выявленные ревью)
- **PRD 41 «✅ ГОТОВО» отозвать → 🟡**: механики GDPR есть, но нет RoPA/DPIA/DPA (F4).
- **PRD 18**: убрать пункт «viewport не задан» — в коде уже корректен.
- **PRD 38**: исправить ссылки DSA (statement of reasons = Art.17, не 16).
- **PRD 14/40**: DSA Art.30 KYBC НЕ откладывать «до потери micro-exemption» — при escrow применяется к traders сразу.

## 5. Рекомендованный порядок закрытия
1. **Фундаменты F1–F14** (большинство — pre-req для денег/доверия/данных).
2. **Юр-гейт F3 + F4 + F5** параллельно (внешние/долгие — начать первыми).
3. Затем области: B (транзакц. ядро) → C (идентичность/анти-фрод) → A-доработки (62 фикс 1996, 31 воронка, 33/32 SEO) → D/монетизация.
4. SEO+Security P0-хардening — финальным этапом (решение пользователя), но F1/F2/F8/F9/F10 из security нужны раньше, т.к. это денежная/анти-фрод основа, а не «косметика».
