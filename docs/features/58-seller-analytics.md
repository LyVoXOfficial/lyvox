## 1. Зачем это (Problem & Outcome)
- **Проблема.** Продавец не понимает, как «работают» его объявления → меньше активности и мотивации улучшать/бустить.
- **Исход.** Дашборд: просмотры, лайки/избранное, контакты, конверсия, рекомендации по улучшению (фото/цена). Мотивирует буст и качество.
- **KPI:** использование дашборда; рост качества листингов; конверсия в буст; ретеншен продавцов.

## 2. Контекст и текущее состояние (As-is)
- Есть `advert_views`, `advert_likes`, favorites, чат-контакты — сырьё; агрегированного дашборда нет.

## 3. Пользователи и сценарии
- **Продавец/бизнес:** «Видеть просмотры/лайки/контакты по объявлениям и что улучшить».

## 4. UX и поведение (детально)
- Дашборд по объявлениям: метрики, тренды, сравнение; подсказки («добавьте фото», «цена выше рынка [[52-ai-fair-price]]»).
- Для Pro — расширенная аналитика.
- Состояния: есть данные/мало данных.

## 5. Логика и правила
- Агрегация просмотров/лайков/избранного/контактов/сделок по объявлению/продавцу.
- Privacy: не раскрывать персональные данные смотрящих (только агрегаты).
- Краевые: новые объявления без данных.

## 6. Данные (Data model)
- Переиспользовать `advert_views`/`advert_likes`/favorites/chat/deals; (опц.) материализованные агрегаты для скорости. RLS: владелец видит свою аналитику.

## 7. API / интеграции
- `/api/seller/analytics` (агрегаты по владельцу); кэш/материализация при росте.

## 8. Безопасность, приватность, комплаенс
- Только агрегаты (не личности зрителей); RLS owner-only; GDPR.

## 9. Аналитика
- Использование дашборда, влияние на качество/буст/ретеншен.

## 10. Доступность и i18n
- Доступные графики (текст-альтернативы); i18n `analytics.*` (NL/FR/EN).

## 11. Тестирование
- Unit: агрегаты. API: analytics + RLS (только своё). e2e: дашборд показывает метрики объявления.

## 12. План внедрения (Rollout)

## 13. Открытые вопросы и решения

## Ревью-требования и sign-off (2026-06-28)

> P2-бонус, но без дедупа просмотров и определения воронки/контакта/сделки дашборд **врёт продавцу**. Применить сквозные F-требования: F6 (sink событий `analytics_events`), F11 (дедуп `advert_views`).

### DATA
- **Метрики стоят на ненадёжном `advert_views` (F11, критично).** Нет дедупа, `anon insert with check(true)` → накрученные просмотры, искажённый знаменатель конверсии. Дедуп: уникальность по `(advert_id, coalesce(user_id, ip_hash, anon_id), date_trunc('day', viewed_at))` ИЛИ отдельная `advert_view_daily`. Сырьё оставить, метрики считать по дедупленному. Закрыть открытый anon-insert (rate-limit + hash IP, не хранить сырой `inet` — GDPR). Один фикс чинит 33/34/37/58.
- **«Контакт» и «сделка» не определены в данных.** Контакт = `contact_started` (старт conversation, событие #15); сделки нет (escrow не запущен). Определить воронку end-to-end: `view → detail_view → contact_started → contact_replied → deal_started → deal_completed`. → F6.
- **Формулы дашборда (зафиксировать в §5):** `contact_rate = unique_contacts / unique_detail_views`; `favorite_rate = favorites / unique_detail_views`; окна 7d / 30d / all с трендом.
- **Materialized `seller_analytics_daily`** (advert_id, seller_id, day, views_unique, likes, favorites, contacts, deals) + refresh cron; RLS owner-only через seller_id — против N+1 при росте (агрегация по продавцу требует JOIN на `adverts.user_id`).
- **Единое определение engagement:** `like` (публичный сигнал) и `favorite` (личный буклет) считаются раздельно — документировать, какой идёт в seller-analytics, какой в taste (PRD 34), иначе рассинхрон метрик.
- **Privacy:** только агрегаты; идентичности зрителей не раскрывать; IP — соль-хеш с TTL (F4/GDPR в аналитике).

**Вердикт:** 🔄 — зависит от дедупа `advert_views` (F11) и определения воронки/контакта/сделки через единый sink (F6); без них дашборд показывает накрученные и неопределённые числа. После их фиксации → ✅.
