# PRD: Интегрированная доставка (Bpost / Mondial Relay, PUDO + локеры)

> **Статус кода:** ⛔ НЕ НАЧАТО
> **Категория:** Не хватает (критическое)
> **Приоритет:** P0
> **Зависит от:** [[10-escrow-safe-deal]] (release по скану доставки), [[11-disputes]] (трекинг как доказательство)
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Без интегрированной доставки нет «shipped-only, tracking-gated escrow»: release нечем триггерить, появляются фейк-курьер фишинг и фейк-трекинг. Ручной ввод адреса/трека — трение и риск.
- **Исход.** Покупатель выбирает пункт выдачи/локер; продавцу генерируется предзаполненный лейбл; статусы перевозчика тянутся автоматически; escrow release привязан к **скану доставки/выдачи**.
- **KPI:** доля сделок с интегрированной доставкой; доля споров «не доставлено» (должна падать); время доставки.

## 2. Контекст и текущее состояние (As-is)
- Нет интеграций перевозчиков, нет генерации лейблов, нет PUDO/локер-выбора, нет таблиц shipments.
- Vinted сильна именно интеграцией с Bpost — это table-stakes для BE.

## 3. Пользователи и сценарии
- **Покупатель:** «Выбрать ближайший пункт Bpost/Mondial Relay или локер; видеть трекинг».
- **Продавец:** «Получить готовый лейбл, не вводить адрес руками; сдать в пункт».
- Happy-path: сделка оплачена → покупатель выбрал пункт выдачи → продавцу лейбл → сдал → трекинг «в пути» → «выдано» → escrow release.

## 4. UX и поведение (детально)
- **Выбор доставки** в форме сделки: домашняя/пункт выдачи/локер; карта/список ближайших точек по индексу.
- **Лейбл продавцу:** PDF/QR, предзаполнен; инструкция сдачи; срок сдачи.
- **Трекинг:** статусы в кабинете сделки и в таймлайне; пуш/almail на ключевых статусах ([[36-notifications]]).
- **Состояния:** лейбл создан → сдано → в пути → прибыло в пункт → выдано → (возврат, если не забрали).
- **Краевые:** не забрал в срок (авто-возврат отправителю), потеря, повреждение (→ [[11-disputes]]).

## 5. Логика и правила
- **Release-триггер escrow** = статус «выдано/доставлено» от перевозчика (не ручной).
- Выбор перевозчика по цене/покрытию/доступности API: bpost, Mondial Relay (старт), затем DPD/DHL.
- Расчёт стоимости/срока; кто платит доставку — настраиваемо в сделке.
- Идемпотентная обработка вебхуков статусов; маппинг кодов перевозчиков в внутренние статусы.

## 6. Данные (Data model)
- `deal_shipments { id, deal_id, carrier, service, label_url, tracking_number, pudo_point_id, status, shipped_at, delivered_at }`
- `carrier_points` (кэш PUDO/локеров) или запрос на лету.
- RLS: участники сделки; service-role server-only. Хранение адресов — минимизация/retention (GDPR).

## 7. API / интеграции
- Внешние API: bpost / Mondial Relay (точки, тарифы, лейблы, трекинг-вебхуки).
- Эндпойнты: `GET /api/shipping/points?postcode`, `POST /api/deals/[id]/label`, `POST /api/webhooks/carrier` (идемпотентно).
- Абстракция «carrier provider», чтобы добавлять перевозчиков без переписывания логики сделки.

## 8. Безопасность, приватность, комплаенс
- Адрес покупателя — минимально необходимый, не показывать продавцу сверх нужного для лейбла; PUDO снижает раскрытие домашнего адреса.
- Анти-фрод: только трекинг от интеграции считается доказательством (бьёт фейк-трекинг).
- GDPR: retention адресов/лейблов.

## 9. Аналитика
- `shipment_label_created/shipped/delivered/returned`, время доставки, доля невыкупа, распределение по перевозчикам/точкам.

## 10. Доступность и i18n
- Доступный выбор точек (поиск, клавиатура, карта с текстовым списком); i18n `shipping.*` (NL/FR/EN), локализованные статусы.

## 11. Тестирование
- Unit: маппинг статусов, расчёт тарифов. API: точки/лейбл/вебхук-идемпотентность (sandbox перевозчика). e2e: выбор точки → лейбл → симулированный трекинг → release. Негатив: невыкуп → авто-возврат.

## 12. План внедрения (Rollout)
- **NEXT** (после базового escrow): Bpost/Mondial Relay PUDO + локеры, release по скану. Затем DPD/DHL. Усилие: M.

## 13. Открытые вопросы и решения
- Стартовый перевозчик (Bpost vs Mondial Relay) и доступ к API/контракт. Кто платит доступ. Сроки невыкупа.

## 14. Критерии готовности (DoD)
- [ ] Покупатель выбирает PUDO/локер; продавцу генерится лейбл.
- [ ] Трекинг тянется автоматически; release escrow привязан к статусу доставки.
- [ ] Вебхуки идемпотентны; адреса минимизированы и под retention.
- [ ] Тесты §11 (sandbox) зелёные.

## Ревью-требования и sign-off (2026-06-28)

### DEV
- **Верификация carrier-вебхука (Critical — денежный триггер, S1):** carrier-вебхуки часто без подписи. `CarrierProvider.parseWebhook(raw, sig)` обязан верифицировать источник, иначе подделка статуса «delivered» = ложный escrow-release. Без верификации статус НЕ считается доказательством release.
- **F1 — идемпотентность carrier-событий (S0):** общий журнал `webhook_events`; т.к. у carrier часто нет уникального `event.id` — составной ключ `carrier:{tracking}:{eventCode}:{ts}`. Повтор → no-op.
- **Маппинг статусов:** таблица `carrier_status_map(carrier, carrier_code, internal_status)` или хелпер `mapStatus` — per-carrier маппинг в внутренний enum (в §5/§6).
- **S1 — абстракция `CarrierProvider`** (`searchPoints/createLabel/parseWebhook/mapStatus`); Bpost/Mondial Relay — конкретные реализации.
- **Правило release (S6):** только internal_status `delivered`/`picked_up` И сделка в `shipped` (оптимистичная блокировка). Невыкуп: `returned` → явный перевод сделки в спорный/refund-флоу (краевой случай довести до денежного решения, не оставлять подвешенным).
- **PII-минимизация адреса (с DEV-уточнением):** адрес доступен только при генерации лейбла, server-side; не дублировать в `deal_events.payload`; хранить в отдельной защищённой таблице/полях с RLS.
- **Поля (дополнить §6):** `deal_shipments.raw_last_event jsonb`, `version int`. Лейбл идемпотентен — один лейбл на сделку (`POST /api/deals/[id]/label`).
- **Rate-limit:** `GET /api/shipping/points` — per-IP (серверный IP, F8/S-1).
- **Тесты (S7):** sandbox-вебхук, повтор события, out-of-order, невыкуп, подделка статуса без валидной подписи → отвергнуто.

### LEGAL
- **GPSR cross-ref (PRD 15):** при доставке traceability товара (производитель/импортёр/EU responsible person) релевантна для new/trader-товаров; для escrow-доставки опасных товаров — takedown/обработка «without undue delay». Кросс-ссылка на PRD 15/38.
- **GDPR адресов:** минимизация (адрес покупателя — только необходимое для лейбла, PUDO снижает раскрытие домашнего адреса) + retention лейблов/адресов по таблице F4; основание Art.6(1)(b).

**Вердикт:** 🔄 — верификация carrier-вебхука (денежный триггер), маппинг статусов, idempotency-ключ без `event.id`, денежный исход невыкупа, минимизация адреса. Выкатывать после базового escrow (PRD 10).
