## 1. Зачем это (Problem & Outcome)
- **Проблема.** Рост на холодном старте дорог; сарафан — дешёвый канал.
- **Исход.** Пользователь приглашает друзей по ссылке; обе стороны получают бонус (буст/скидка/protection-fee-кредит) после квалифицирующего действия.
- **KPI:** доля приглашающих, viral-коэффициент, стоимость привлечения через рефералов, фрод-rate рефералов.

## 2. Контекст и текущее состояние (As-is)
- Нет. Бизнес-инвайты команды есть ([[40-business-accounts]]) — другой механизм.

## 3. Пользователи и сценарии
- «Поделиться реф-ссылкой → друг регистрируется/совершает действие → оба получают бонус».

## 4. UX и поведение (детально)
- Реф-ссылка/код в профиле; шаринг ([[57-social-sharing]]).
- Прогресс: приглашён/зарегистрирован/квалифицирован/вознаграждён.
- Состояния: ожидание/начислено/отклонено (фрод).

## 5. Логика и правила
- Квалифицирующее событие (верификация/первая сделка), а не просто регистрация (анти-фарм).
- Анти-абуз: уникальность по verified identity ([[30-identity-account]]), лимиты, детект self-referral/мульти-аккаунтов.
- Краевые: отмена сделки после награды (клоубэк), фрод-кольца.

## 6. Данные (Data model)
- `referrals { id, referrer_id, referee_id, code, status, qualified_at, reward_id }`. RLS owner-only. Аудит начислений.

## 7. API / интеграции
- `/api/referrals` (код/статус), начисление через billing/benefits.

## 8. Безопасность, приватность, комплаенс
- Анти-фрод рефералов (главный риск); прозрачные правила; GDPR.

## 9. Аналитика
- Viral-коэффициент, конверсия квалификации, фрод-rate, стоимость награды/привлечение.

## 10. Доступность и i18n
- i18n `referral.*` (NL/FR/EN).

## 11. Тестирование
- Unit: квалификация/клоубэк/анти-абуз. API: код/начисление. e2e: приглашение → квалификация → награда.

## 12. План внедрения (Rollout)

## 13. Открытые вопросы и решения

## Ревью-требования и sign-off (2026-06-28)

> Единственный PRD зоны с условным ✅ от LEGAL — низкий риск **при соблюдении 3 условий**. Главный риск — реферал-фрод (SEC). Применить сквозные F-требования: F8 (серверный IP), F10 (`itsme_sub` uniqueness — блокирующая зависимость).

### LEGAL (✅ при условиях)
1. **Никакой платформенной рассылки инвайтов.** Если LyVoX сам рассылает приглашения по введённым email/контактам — это **директ-маркетинг без согласия получателя** (ePrivacy Art.13 / CDE Book VI Art. VI.110) = нарушение. Зафиксировать: только **self-share** пользователем своей ссылки; запрет импорта контактов/рассылки платформой.
2. **Прозрачные Referral Terms.** Условия квалификации/награды/сроков/лимитов/clawback — доступны до участия (UCPD, не misleading). Основание отказа «фрод» + право оспорить (нельзя произвольно лишать обещанной награды); право платформы изменить с уведомлением. Clawback при отмене сделки — договорное условие, в правилах **заранее**, не постфактум.
3. **GDPR-нотис для identity-dedup анти-фрода.** Связывание аккаунтов по verified identity = профайлинг → нотис + основание (legitimate interest + LIA). Art.22-гарантия при автоматическом отказе в награде.

### SEC
- **Реферал-фрод / self-referral / мульти-аккаунт (главный риск фичи).** Без серверной уникальности по verified identity (itsme `sub`) — фарм через throwaway-аккаунты. **Блокирующая зависимость (F10/S-12):** колонки `itsme_sub` в схеме НЕТ (миграция `20251110000000_itsme_fields.sql` = только `itsme_verified`+`itsme_kyc_level`) → анти-абуз рефералов не на чём строить. Квалификация = verified-tier действие, не регистрация.
- **Clawback атомарно** при отмене/возврате сделки; награда «pending» до невозвратности.
- **Фрод-кольца:** серверный IP (F8, `getClientIp` не доверять `x-forwarded-for`) + device-кластеризация для детекта взаимных колец; лимиты наград/период.
- **referrer-привязка фиксируется сервером** при регистрации (RLS owner-only, не клиент-editable) — анти-IDOR/самоназначение referrer.
- **Реварды как денежный переход** (protection-fee-кредит/буст) — через тот же authorization+аудит, что billing (S-5/S-7 column-lock RLS).

**Вердикт:** ✅ с условием — чисто при трёх LEGAL-условиях (нет платформенной рассылки, прозрачные Referral Terms+clawback, GDPR-нотис dedup). SEC: блокирующая зависимость от `itsme_sub` uniqueness (F10) и серверного IP (F8) до запуска анти-абуза. P2 — строить после доказанного trust-loop.
