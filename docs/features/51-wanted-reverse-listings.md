# PRD: Обратные объявления «Ищу» (wanted / wishlist-спрос)

> **Статус кода:** ⛔ НЕ НАЧАТО
> **Категория:** Бонус
> **Приоритет:** P2
> **Зависит от:** [[33-search]], [[34-discovery-extras]] (матчинг с saved-search), [[36-notifications]]
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** На холодном старте товаров мало — покупатели уходят. «Ищу»-объявления дают ликвидность спроса, когда предложения ещё мало.
- **Исход.** Покупатель публикует запрос (что ищет, бюджет, локация); продавцы видят спрос и матчатся; при появлении подходящего товара — алерт обеим сторонам.
- **KPI:** число «ищу»-постов; матч-rate; конверсия матча в контакт; вклад в ликвидность.

## 2. Контекст и текущее состояние (As-is)
- Нет. Есть saved-searches (близкий механизм) и поиск — переиспользуем матчинг.

## 3. Пользователи и сценарии
- **Покупатель:** «Опубликовать “ищу iPhone до 400€ в Генте” → получать совпадения».
- **Продавец:** «Видеть спрос и предложить свой товар».

## 4. UX и поведение (детально)
- Создание «ищу»: категория/атрибуты/бюджет/локация/описание.
- Лента «Спрос» для продавцов; матчинг с существующими/новыми листингами; алерт.
- Состояния: активно/исполнено/истекло.

## 5. Логика и правила
- Матчинг = обратный saved-search (запрос против листингов и наоборот).
- Анти-спам: лимиты на число «ищу», модерация ([[38-moderation-reports]]).
- Краевые: слишком общий запрос, устаревший спрос.

## 6. Данные (Data model)
- `wanted_requests { id, user_id, category_id, attributes jsonb, budget_max, location, status, expires_at }`. RLS owner-only на управление; публичная лента спроса. Retention.

## 7. API / интеграции
- `/api/wanted` (CRUD), матч-движок (переиспользует search/saved-search), нотификации.

## 8. Безопасность, приватность, комплаенс
- Не раскрывать лишние контакты; модерация; GDPR-retention.

## 9. Аналитика
- Число «ищу», матч-rate, контакт из матча, вклад в ликвидность.

## 10. Доступность и i18n
- i18n `wanted.*` (NL/FR/EN).

## 11. Тестирование
- Unit: матчинг. API: CRUD + матч. e2e: «ищу» → появление товара → алерт.

## 12. План внедрения (Rollout)
- **LATER (но раннее по ценности для cold-start):** MVP без атрибут-матчинга, затем точнее. Усилие: M.

## 13. Открытые вопросы и решения
- Публичность спроса vs приватность. Срок жизни запроса. Лимиты.

## 14. Критерии готовности (DoD)
- [ ] Создание/управление «ищу»; лента спроса.
- [ ] Матчинг + алерты обеим сторонам.
- [ ] Модерация/лимиты; i18n; тесты §11 зелёные.
