# PRD: Геймификация trust-уровня

> **Статус кода:** 🟡 ЧАСТИЧНО (badges_awarded есть; прогресс-механики нет)
> **Категория:** Бонус
> **Приоритет:** P2
> **Зависит от:** [[37-trust-reviews]], [[30-identity-account]] (верификация), [[10-escrow-safe-deal]] (сделки)
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Пользователю непонятно, как повысить доверие/«статус продавца»; нет мотивации проходить верификацию и завершать сделки честно.
- **Исход.** Прогресс «до Verified seller / Trusted seller»: понятные шаги (верифицируй телефон/itsme, заверши N сделок, получи отзывы), бейджи-уровни, видимый прогресс. Подталкивает к доверенным действиям.
- **KPI:** доля прошедших верификацию, рост завершённых сделок, доля достигших уровней.

## 2. Контекст и текущее состояние (As-is)
- `badges_awarded` есть; нет прогресс-механики/чек-листа/уровней, привязанных к trust-сигналам ([[37-trust-reviews]]).

## 3. Пользователи и сценарии
- **Продавец:** «Вижу, что нужно сделать, чтобы стать Trusted seller, и мой прогресс».

## 4. UX и поведение (детально)
- Чек-лист доверия в профиле: верификация e-mail/phone/itsme, первая сделка, N отзывов и т.д.
- Уровни/бейджи (New → Verified → Trusted → Top), прогресс-бар, разблокировки (напр. выше лимиты escrow/быстрее payout).
- Состояния: уровни и оставшиеся шаги.

## 5. Логика и правила
- Прогресс **только из проверяемых событий** (верификация, завершённые сделки, отзывы) — без накрутки ([[37-trust-reviews]]).
- Разблокировки уровней привязать к реальным привилегиям (доверие, не косметика).
- Краевые: откат уровня при споре/нарушении; новичок.

## 6. Данные (Data model)
- `badges_awarded` + правила уровней (конфиг); агрегаты из trust-сигналов/сделок. RLS: публичный бейдж, owner видит чек-лист.

## 7. API / интеграции
- `/api/trust/progress` (чек-лист/уровень); интеграция с верификацией/сделками/отзывами.

## 8. Безопасность, приватность, комплаенс
- Анти-накрутка (только проверяемые события); прозрачность правил; без тёмных паттернов.

## 9. Аналитика
- Прохождение шагов, достижение уровней, влияние на верификацию/сделки.

## 10. Доступность и i18n
- Доступные прогресс-индикаторы; i18n `trust_level.*` (NL/FR/EN).

## 11. Тестирование
- Unit: расчёт уровня/прогресса, откат при нарушении. e2e: верификация/сделка → прогресс/уровень растёт.

## 12. План внедрения (Rollout)
- **LATER:** после trust_score из сделок ([[37-trust-reviews]]). Усилие: M.

## 13. Открытые вопросы и решения
- Пороги уровней и привилегии. Откат при споре. Что показывать публично.

## 14. Критерии готовности (DoD)
- [ ] Чек-лист доверия + уровни/бейджи из проверяемых событий.
- [ ] Привилегии уровней привязаны к доверию; откат при нарушении.
- [ ] i18n; тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

Ревью: критичных замечаний именно по этому PRD нет; применить сквозные F-требования где релевантно. Главная зависимость — от формулы trust из PRD 37.

### DATA / SEC (наследуется)
- **Зависит от явной trust-формулы (F14, ⛔ у PRD 37).** Уровни/прогресс должны опираться на разложенные субскоры из `trust_score_components` (identity / deals / reviews / response / disputes / risk) с весами — иначе «прогресс» необъясним. Пока формулы PRD 37 нет, этот PRD не может корректно считать пороги уровней.
- **Анти-накрутка (только проверяемые события — есть в §5, усилить F14):** прогресс из verification + завершённых escrow-сделок + валидных отзывов; reviews-стэкинг по продавцу закрывается на уровне 37 (`unique(subject_id, reviewer_id)` + вес ревьюера); просмотры/лайки **не** засчитывать в уровень (накручиваемы, F11).
- **Портативность уровня по verified identity (`sub`), не по `user_id` (F10/S-12).** При пере-регистрации с тем же itsme `sub` уровень не сбрасывается и не фармится заново — требует `itsme_sub` uniqueness (которой в схеме нет).
- **Art.22-гарантия (LEGAL-смежно):** если уровень/откат блокирует значимое действие (лимиты escrow, скорость payout) — нотис о tiering + право человеческого пересмотра (как в PRD 30).
- **Откат при споре/нарушении** (есть в §5) — атомарно из проверяемого dispute-исхода, иммутабельный аудит изменения уровня.

**Вердикт:** 🔄 — блокирующая зависимость от формулы trust (PRD 37, F14) и портативности по `sub` (F10); пороги уровней нельзя финализировать до них. После — самостоятельных блокеров нет → ✅.
