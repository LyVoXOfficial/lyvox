> **Статус кода:** 🟡 ЧАСТИЧНО (механика есть, нужна доработка визуала и жестов)

## 1. Зачем это (Problem & Outcome)

**Проблема.** Классический поиск-список требует усилий: пользователь должен сам формулировать запрос, листать, открывать карточки. Для second-hand с импульсной покупкой это высокий барьер, особенно на мобильном. Лента-свайпы (как в Tinder) превращает просмотр в быстрый дофаминовый цикл «фото → решение за 1 жест», что: (а) увеличивает глубину просмотра и лайки, (б) собирает сигнал предпочтений почти без трения, (в) повышает шанс контакта с продавцом, (г) — на холодном старте — помогает показать имеющееся предложение максимально привлекательно.

**Исход.** Discover становится вторым по важности каналом дискавери после поиска и главным на мобильном: высокий % сессий доходит до ≥20 свайпов, заметная доля свайпов конвертируется в лайк/избранное/контакт.

**Связь со стратегией.** Прямо бьёт по ликвидности (главный риск): растит вовлечённость покупателя и сигнал спроса для продавца. Не подменяет trust-слой, но усиливает воронку к контакту, где уже работают trust-гейты и чат-антифрод.

**KPI (предложение):**
- Discover swipe-through: медиана свайпов за сессию ≥ 20.
- Like-rate (свайп вправо / все свайпы): 15–30% (ориентир, калибруем).
- Действие из ленты → старт чата: ≥ 3% от сессий с Discover.
- Возврат в Discover (D7 retention среди тех, кто его открыл): растёт от базовой.
- Доля «отмен последнего свайпа» < 8% (показатель случайных свайпов; высокий → менять пороги).

## 2. Контекст и текущее состояние (As-is)

Реализовано (файлы — фактические):
- **Экран:** `apps/web/src/app/discover/page.tsx` — рендерит `SwipeDeck`, дека сама грузит первую страницу клиентски через `/api/search`.
- **Дека:** `apps/web/src/components/discover/SwipeDeck.tsx` (~414 строк) — указательные жесты (pointer), пороги, дозагрузка, лист действий.
- **Карточка:** `apps/web/src/components/discover/SwipeCard.tsx` — презентационная, фото + scrim (title/price/location/likeCount), сердечко-избранное, лейбл-подсказка при перетаскивании. **Имя продавца не показывается** (идентичность скрыта до контакта).
- **Данные деки:** `apps/web/src/lib/discover/deck.ts` — `DeckCard`, маппинг из `/api/search`, тематические `DROPS`, `rerankByTaste`, `filterUnseen`.
- **Taste-модель:** `apps/web/src/lib/taste.ts` — анонимная, на `localStorage`, веса по dimension (категория/продавец/локация/ценовой бэнд) из сигналов like (+1) / favorite (+2) / pass (−1); `scoreCard()` суммирует. Cold start = всё 0 = исходный порядок. Уважает `cookieConsent('functional')`.
- **Просмотренное:** `apps/web/src/lib/seenAdverts.ts` (`addSeenAdverts`, `isSeen`).

**Текущие жесты:**
| Жест | Действие | Код |
|---|---|---|
| Свайп вправо (dx > 110) | Like (+taste, addLike под `requireTrust('auth')`) | `doLike` |
| Свайп влево (dx < −110) | Pass (−taste) | `doPass` |
| Свайп вверх (dy < −110) | Лист действий (открыть / написать / предложить цену) | `doAct` → Sheet |
| Тап (сдвиг < 8px) | Открыть объявление `/ad/[id]` | `onPointerUp` |
| Клавиши ←/→/↑/Enter | Pass / Like / Действия / Открыть | `onKeyDown` |
| Кнопки внизу | Pass / Действия / Like | три круглые кнопки |
| Сердечко на карточке | Избранное (независимо от свайпа) | `FavoriteToggle` |

**Чего нет / что слабо (объекты доработки):**
1. **Нет свайпа вниз** — четвёртое направление простаивает.
2. **Нет отмены последнего свайпа** (rewind) — случайный свайп необратим (иконка `RotateCcw` занята под «сброс предпочтений», не под undo).
3. **Нет анимации вылета карточки** — на коммите очередь просто срезается (`queue.slice(1)`), карточка не «улетает».
4. **Нет коммита по «флику»** (скорость) — только по дистанции 110px; быстрый короткий флик не срабатывает.
5. **Нет тактильной отдачи (haptics)** на пересечении порога/коммите.
6. **Карточка показывает только 1 фото** — нет пролистывания галереи внутри карточки и trust-сигналов (бейдж verified, «новое»).
7. **Нет онбординг-подсказки** (coach-mark) — новый пользователь не знает, что делает каждый жест.
8. **Нет выбора схемы управления / упрощённого и кнопочного режимов** в настройках.
9. **Нет аналитических событий** по свайпам.

## 3. Пользователи и сценарии (User stories)

- **Guest / Registered:** «Хочу быстро листать товары жестами и отмечать интересное, не открывая каждую карточку».
- **Registered → Verified:** «Свайпнул вверх → хочу сразу написать продавцу или предложить цену» (срабатывает trust-гейт `verified`).
- **Любой:** «Случайно свайпнул не туда → хочу вернуть карточку».
- **Пользователь с укачиванием от анимаций:** «Хочу отключить движение» (`prefers-reduced-motion`).
- **Пользователь, которому 4 жеста сложны:** «Хочу простой режим только лайк/пропуск».
- **Доступность (скринридер/моторика):** «Хочу управлять кнопками и с клавиатуры».

Happy-path: открыл Discover → видит карточку → свайпает вправо/влево несколько раз → на интересном свайпает вверх → выбирает «Написать» → проходит trust-гейт → попадает в чат.

## 4. UX и поведение (детально)

### 4.1 Итоговая модель жестов (схема «Стандарт» — по умолчанию)

| Направление | Действие | Сигнал taste | Trust-гейт | Визуал |
|---|---|---|---|---|
| **Вправо →** | **Нравится** (мягкий позитив): записать like, поставить лайк объявлению | +1 | `auth` (лайк ставится после) | Зелёный штамп «НРАВИТСЯ» + зелёное свечение края |
| **Влево ←** | **Пропустить** (мягкий негатив) | −1 | нет | Красный штамп «ПРОПУСТИТЬ» + красное свечение |
| **Вверх ↑** | **Действия** (высокий интент): лист с «Открыть / Написать / Предложить цену / В избранное» | 0 (нейтрально; сильный позитив запишется при реальном действии) | `verified` на чат/оффер | Синий штамп «ДЕЙСТВИЯ» + синее свечение |
| **Вниз ↓** *(новое)* | **Меньше такого** (сильный негатив + персонализация): чипы-причины «Дорого / Не та категория / Уже есть / Скрыть продавца» | −2 (плюс mute категории/продавца при выборе) | нет | Серо-фиолетовый штамп «МЕНЬШЕ ТАКОГО» |
| **Тап** | Открыть объявление `/ad/[id]` | 0 | нет | — |
| **Двойной тап / сердечко** | В избранное (favorite) | +2 | `auth` | Анимация сердца |
| **Зоны фото (лево/право край)** *(новое)* | Листать фото внутри карточки (если фото >1) | 0 | нет | Точки-индикаторы сверху |

**Почему именно так (обоснование).**
- Горизонталь = быстрый лёгкий триаж (как в Tinder — нулевой когнитивный порог). Это дофаминовый цикл, который кормит taste и растит лайки.
- Вверх = «деньги»: высокий интент ведёт к контакту/офферу. Оставляем как есть — это конверсионный жест.
- Вниз = недостающее четвёртое направление. Делаем его **сильным негативом + инструментом персонализации** (а не дублем «пропуска»). Это резко улучшает релевантность ленты и даёт пользователю чувство контроля. Причина-чип «Скрыть продавца» — ещё и анти-спам/анти-фрод сигнал.
- Отмену последнего свайпа (rewind) даём **бесплатно** (в Tinder это платно) — для trust-бренда важнее доверие, чем монетизация на ошибках.

### 4.2 Анатомия карточки (визуал)

Контейнер: соотношение 3:4, скруглённые углы (`rounded-2xl`), мягкая тень, full-bleed фото.

Сверху:
- слева — индикаторные точки галереи (если фото > 1);
- справа — сердечко «избранное» (overlay) и **бейдж «Verified»** продавца (если `sellerVerified`), значок «Новое» для свежих (< 48 ч).

Снизу — градиентный scrim (тёмный к прозрачному), на нём:
- заголовок (1 строка, truncate);
- цена (крупно), локация (с иконкой), счётчик лайков;
- маленькая строка trust-сигналов: «Проверенный продавец» / «Безопасная сделка доступна» (когда появится escrow — см. [[10-escrow-safe-deal]]).

Имя/контакты продавца **не показываем** (идентичность скрыта до контакта — сохраняем текущее правило).

Поверх (во время перетаскивания) — **штамп-подсказка** направления (цвет+текст) и цветное свечение края, плавно проявляющиеся по мере приближения к порогу (прозрачность ∝ дистанции).

Под декой — три (в «Стандарт» — пять) круглых кнопки: **Пропустить · Меньше · Действия · Отмена · Нравится** (порядок и набор зависят от режима, см. 4.5).

### 4.3 Анимации и тактильность

- **Drag:** карточка следует за пальцем, лёгкий поворот `rotate(dx * 0.04deg)` (есть). Параллакс штампа.
- **Коммит:** карточка **улетает** за экран в сторону жеста за ~250–300 мс с ускорением (добавить — сейчас нет), затем удаляется из очереди. Карточки за ней плавно «всплывают» (scale/opacity).
- **Snap-back:** не достиг порога → пружинный возврат в центр (есть базово через transition).
- **Порог коммита:** дистанция ≥ 110px **ИЛИ** «флик» — скорость > порога при меньшей дистанции (добавить velocity).
- **Haptics (мобайл):** короткая вибрация при пересечении порога и при коммите (`navigator.vibrate`, только если поддержано и не reduced-motion). Отключаемо в настройках.
- **prefers-reduced-motion:** без поворотов/вылетов — мгновенная смена + лёгкий fade (учтено в коде, расширить на вылет).

### 4.4 Состояния экрана

- **Loading (первая загрузка):** скелет-карточка (мерцающий прямоугольник 3:4) + неактивные кнопки.
- **Пусто (конец деки):** заголовок + текст + кнопки «Сменить подборку (Drop)» / «В ленту». (есть, расширить — предложить сменить Drop).
- **Ошибка:** заголовок + «Повторить». (есть)
- **Offline:** баннер «Нет сети, показываем кэш» + при коммите действий — очередь оффлайн-сигналов (taste пишется локально, лайк/чат откладываются до сети).
- **Конец каталога для Drop (`done`):** мягко предложить другой Drop или «расширить радиус» (когда появится гео).

### 4.5 Решение: «Стандарт сразу» или «выбор пользователя»?

**Принятое решение (с обоснованием):**
1. **По умолчанию — схема «Стандарт» (4 направления), включена для всех, без выбора при онбординге.** Любой обязательный выбор на входе = трение = риск для холодного старта. Пользователь сразу попадает в рабочую ленту.
2. **Первый визит — одноразовый coach-mark** (полупрозрачный оверлей со стрелками и подписями: → Нравится, ← Пропустить, ↑ Действия, ↓ Меньше, тап — открыть). Закрывается тапом «Понятно», больше не показывается (флаг в `localStorage` `lyvox:discover:coach`). Повторно открывается кнопкой «?» в шапке Discover.
3. **Выбор — опционально, в Настройках → Discover** (не на входе):
   - **Режим управления:** `Standard` (4 направления, дефолт) · `Simple` (только вправо/влево + тап; вверх/вниз выключены — для тех, кому 4 жеста сложны) · `Buttons-only` (свайпы выключены, всё кнопками — доступность/моторика).
   - **Тумблеры:** haptics on/off; «спрашивать причину при свайпе вниз» on/off; «подтверждать действия из ленты» on/off.
   - (Позже) «левша»: зеркалирование расположения кнопок.
4. Настройка хранится в профиле (`profiles.discover_prefs` JSONB) с локальным кэшем; для гостя — только `localStorage`.

Итого: **дефолт без трения + мощь для тех, кто хочет настроить + доступность из коробки.**

### 4.6 Drops (тематические подборки)
Оставляем чипы Drops сверху (`just_listed`, `under_50`, `verified`, `cheapest`). Roadmap: добавить гео-Drop «Рядом» (когда будет геопоиск), «По твоему вкусу» (если taste накопил сигнал), персональные Drops по категории интереса.

## 5. Логика и правила (Business logic)

- **Источник карточек:** `/api/search` по `drop.query` постранично (`PAGE_SIZE=24`), дозагрузка при остатке < `LOW_WATER=5` (есть). Дедуп по `id`, фильтр `filterUnseen`, rerank `rerankByTaste`.
- **Пороги:** `THRESH_X=110`, `THRESH_Y=110`, `TAP_MOVE=8` (есть). Добавить velocity-порог (например, > 0.6 px/мс).
- **Сигналы taste:** like +1, favorite +2, pass −1, **down/«меньше» −2** (новое), действие-контакт +2 (новое — записывать при реальном старте чата/оффера). Mute продавца/категории при причине «скрыть» — добавить отрицательный «потолок» или флаг исключения в taste/seen.
- **Trust-гейты:** лайк → `requireTrust('auth')`; чат/оффер → `requireTrust('verified')` (есть, сохраняем).
- **Undo (новое):** буфер последних N=3 свайпов (карточка + тип сигнала). Отмена: вернуть карточку в начало очереди, **откатить taste-сигнал**, снять `seen`, отменить лайк если был поставлен. Лимит и таймаут (например, отмена доступна 5 сек или до следующего свайпа — решить, см. §13).
- **Идемпотентность лайка:** повторный лайк той же карточки не плодит дублей (на стороне `/api/likes`).
- **Гость:** свайпы и taste работают локально; like/favorite/chat поднимают trust-гейт (логин/верификация).

## 6. Данные (Data model)

В основном переиспользуем существующее. Новое/изменения:
- **`profiles.discover_prefs JSONB`** (новое поле, default `{}`): `{ mode: 'standard'|'simple'|'buttons', haptics: bool, ask_reason_down: bool, confirm_actions: bool }`. Миграция `YYYYMMDDHHMMSS_discover_prefs.sql`. RLS: владелец читает/пишет своё (как у прочих profile-полей; учесть `lock_profiles_columns`).
- **Сигнал «меньше такого» / mute:** на этапе MVP — локально (`taste` отрицательный вес + `seenAdverts`/`localStorage` список mute по `seller:`/`cat:`). Если нужен серверный кросс-девайс mute — таблица `user_hidden { user_id, target_type ('seller'|'category'|'advert'), target_id, created_at }` + RLS owner-only (вынести в roadmap, не блокер).
- **Аналитика свайпов:** события (см. §9) — в существующий механизм логирования/`advert_views` не мешаем; для свайпов завести лёгкую таблицу или внешнюю аналитику (см. §9).
- **Taste остаётся клиентским** (localStorage), не БД — это осознанно (приватность, нулевая нагрузка). Серверная персонализация — отдельный roadmap.

GDPR: `discover_prefs` и серверный `user_hidden` попадают в экспорт/удаление аккаунта ([[41-gdpr-legal]]).

Все мутации — серверная авторизация + RLS; `supabaseService()` только server-only.

## 8. Безопасность, приватность, комплаенс

- **Идентичность продавца скрыта** в карточке (правило сохраняется) — снижает off-platform увод и таргетинг.
- **Trust-гейты** на лайк/чат/оффер не обходятся через ленту (используется общий `requireTrust`).
- **Приватность taste:** локально, под `functional`-consent; «Сброс предпочтений» доступен (есть).
- **Анти-абуз:** «Скрыть продавца» как сигнал; rate-limit на старт чата (есть на `/api/chat/*`).
- **GDPR:** новые префы/mute — в экспорт/удаление.
- **A11y-комплаенс:** кнопочный режим обязателен (см. §10).

## 10. Доступность и i18n

**A11y:**
- Полное управление кнопками (`Buttons-only`) и с клавиатуры (←/→/↑/↓/Enter/U=undo).
- Все кнопки имеют `aria-label` (есть для трёх; добавить для «Меньше» и «Отмена»).
- Тач-таргеты ≥ 44px (кнопки 48–56px — ок).
- `prefers-reduced-motion` отключает вылеты/повороты/haptics.

## 13. Открытые вопросы и решения

**Решено:**

## Ревью-требования и sign-off (2026-06-28)

> Каркас отличный (4 направления, бесплатный undo, режимы, reduced-motion, богатая телеметрия), но конверсионно-опасны два пробела: анти-социальное доказательство `likeCount` в карточке и ложный empty-state на старте. Применить сквозные F-требования: F6 (единый sink `analytics_events`), F11 (дедуп `advert_views`).

### UX
- **Анти-социальное доказательство `likeCount` (High, сквозное правило).** На холодном старте «👍 0» / «👍 1» на каждой карточке = инверсия доверия в самом конверсионном месте. **Показывать счётчик только при пороге** `likeCount >= N` (старт N=3–5); ниже порога — **не рендерить число вовсе** (не «0»). То же для «просмотров». Зафиксировать в §4.2 и DoD.
- **Ложный empty-state на старте — запретить логикой и копирайтом.** До первой загрузки показывать скелет-карточку, НЕ текст «Вы всё просмотрели» (флаг `hasLoadedOnce`). Empty-state в конце деки переписать дружелюбно: «Вы посмотрели всё свежее» + CTA «Показать снова» (сброс `seenAdverts`, не только taste) / «Сменить подборку» / «Сохранить поиск».
- **Бизнес-гейт включения Discover по объёму каталога.** На каталоге 4–10 карточек метрики шумные, а свайп-режим = ускоренный путь к ложному empty-state. Включать канал (фиче-флаг `discover_v2`) только при `каталог >= порог` (рекоменд. ≥100 активных карточек в гео/категории). В §12 Rollout.
- **Жестовый конфликт «вниз».** Свайп вниз = «Меньше такого» конфликтует с системным «закрыть/скролл». Явно: в зоне деки отключить системный скролл (`touch-none` уже есть), pull-to-refresh/overscroll деки выключить (особенно iOS Safari). В §4.3 + тесты.
- **Убрать двойной тап как «в избранное»** — конфликтует с одинарным «открыть» (залипание из-за задержки распознавания). Избранное — только сердечко (≥44px) и свайп-вверх-лист; одинарный тап = открыть без задержки. В §4.1.
- **Чипы-причины при свайпе вниз** — не модалка-прерывание: первые N раз спрашиваем, потом «вниз» молча даёт −2 и тихо показывает undo-снэк «Скрыли. Почему? [чипы]», не блокируя следующий свайп. Добавить нейтральный 5-й чип «Просто не интересно».
- **Якорь безопасности в листе действий (свайп вверх):** строка «Общайтесь и платите только в LyVoX — так безопаснее» — снижает пост-фактум тревогу в точке интента. Trust-гейт: не «403», а «Чтобы написать продавцу, подтвердите телефон — это защищает обе стороны».
- **Undo affordance:** кнопка «Вернуть карточку» всегда видна и активна когда буфер непуст, задизейблена (не скрыта) когда пуст.
- **a11y:** focus-ring на всех элементах деки/чипах; лист действий и чипы-причины — фокус-trap + Esc + возврат фокуса; coach-mark доступен скринридеру и с клавиатуры (унаследовать из движка 61); `aria-live` объявляет смену карточки только при финальном коммите, не при drag; контраст текста на scrim ≥4.5:1 гарантировать плашкой/градиентом (не «проверить»).

### DATA
- **Нет приёмника/схемы хранения событий — KPI неизмеримы (F6).** §6 «лёгкая таблица или внешняя аналитика» не решено. Завести единый sink `analytics_events(id, session_id, user_id|anon_id, event_name, props jsonb, ts, client_dedup_key)` (переиспользуемый в 33/58); батч-инсерт; RLS insert свои, чтение — только аналитика/service. Без него медиана свайпов / like-rate / undo-rate не посчитать.
- **Добавить `session_id` во все `discover_*`** — иначе нельзя собрать воронку open→N свайпов→action→chat на уровне сессии (`position_in_session` есть, `session_id` нет).
- **Добавить `dwell_ms` в `discover_swipe`** — время до решения, ключевой сигнал качества/случайности свайпа (связан с undo-rate KPI).
- **`client_dedup_key` (uuid на клиенте)** для идемпотентности при offline-ресинке (§4.4) — иначе дубли свайпов исказят метрики.
- **Серверный лог taste-сигналов** (даже при клиентском rerank): `taste_signals(user_id|anon_id, advert_id, signal, weight, ts)` под `functional`-consent — для офлайн-оценки качества рекомендаций (precision@k, like-rate uplift A/B). Связь с PRD 34.
- **Воронка-определение в §9:** `discover_open → discover_swipe (count≥N) → discover_action_message → chat_started`; like-rate = `swipe[dir=like]/swipe[all]` за сессию.
- **Дедуп `advert_views` (F11)** — KPI Discover опираются на просмотры; накрутка течёт в ранжирование/trust/аналитику. Общий фикс с 33/34/37/58.
- **KPI like-rate 15–30% при пустом каталоге недостижим** — согласуется с UX бизнес-гейтом включения по объёму каталога; пометить метрики как «ранние» до достижения порога ликвидности.

**Вердикт:** 🔄 — каркас силён; обязательны: порог `likeCount` + `hasLoadedOnce` + единый sink событий с `session_id`/`dwell_ms`/dedup-ключом, бизнес-гейт каталога, убрать double-tap, a11y (focus-trap sheet, доступный coach-mark, focus-ring). После правок → ✅.
