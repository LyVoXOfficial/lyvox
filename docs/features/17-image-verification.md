# PRD: Проверка фото (reverse-image, хеш, краденые фото)

> **Статус кода:** ⛔ НЕ НАЧАТО (AI-модерация работает по тексту объявления, не по фото-верификации)
> **Категория:** Не хватает (усиление анти-фрода)
> **Приоритет:** P1
> **Зависит от:** [[38-moderation-reports]] (очередь/решения), [[31-listing-creation]] (загрузка медиа)
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Скамеры берут чужие фото (украденные/из интернета), переиспользуют одни фото в множестве объявлений, постят контрафакт/краденое. Сейчас нет хеш-матчинга/reverse-image.
- **Исход.** Дубликаты и украденные фото детектируются как фрод-сигнал; контрафакт/опасное — на модерацию; повторное использование фото между аккаунтами кластеризуется.
- **KPI:** доля задержанных bait/scam-листингов по фото-сигналу; падение жалоб «фейк».

## 2. Контекст и текущее состояние (As-is)
- Медиа: загрузка signed-storage, статус модерации (`/api/media/*`); AI-moderation Edge Function скорит текст/метаданные объявления; **нет** image-hash/reverse-image/perceptual matching.

## 3. Пользователи и сценарии
- **Система:** «При загрузке фото — вычислить перцептивный хеш, сверить с базой; флаг при совпадении/находке в вебе».
- **Модератор:** «Вижу фото-сигналы (дубликат/украдено/контрафакт) в очереди».
- **Покупатель:** косвенно — меньше фейков в ленте.

## 4. UX и поведение (детально)
- **При публикации:** фоновая проверка; при сильном совпадении — мягкая задержка/флаг на модерацию (не блок «в лоб», чтобы не бить честных).
- **Модерация:** карточка с найденными совпадениями (другие объявления с тем же фото, источник в вебе).
- **Продавцу:** при флаге — запрос «добавьте живое фото / фото с кодовым словом».
- **Состояния фото:** ok / подозрение (дубликат) / подозрение (украдено) / на ревью.

## 5. Логика и правила
- **Perceptual hash (pHash)** всех загружаемых фото → сверка между листингами (кластеры повторного использования).
- **Reverse-image / stolen-photo** — через off-the-shelf API (не хэндроллить).
- **Контрафакт** — сигнал для AI/модерации; быстрый takedown ([[38-moderation-reports]] — DSA Art.16) + политика повторных нарушителей (Art.23).
- Пороги совпадения; защита от ложных срабатываний (стоковые/типовые фото).

## 6. Данные (Data model)
- `media_hashes { media_id, advert_id, phash, source_check_status, created_at }`
- `media_flags { id, media_id, flag_type ('duplicate'|'stolen'|'counterfeit'), score, action, created_at }`
- Индексы для быстрого hash-поиска; RLS (модерация/админ); retention.

## 7. API / интеграции
- Хук в `/api/media/complete` (после загрузки) → вычислить pHash + (опц.) reverse-image API.
- Внешний reverse-image/stolen-photo сервис; интеграция с очередью модерации.

## 8. Безопасность, приватность, комплаенс
- Обработка изображений — на сервере; не хранить лишнего; GDPR-retention.
- DSA: быстрый takedown + statement of reasons; повторные нарушители.

## 9. Аналитика
- `image_hashed`, `image_flag_duplicate/stolen/counterfeit`, доля листингов с флагом, исходы модерации.

## 10. Доступность и i18n
- Понятные сообщения продавцу при флаге; i18n `media.flag.*` (NL/FR/EN).

## 11. Тестирование
- Unit: pHash, порог совпадения, ложные срабатывания на стоке. API: загрузка дубликата → флаг. e2e: украденное фото → ревью.

## 12. План внедрения (Rollout)
- **LATER:** pHash-дубликаты (дёшево, свои силы) → reverse-image/stolen (API) → контрафакт. Усилие: H.

## 13. Открытые вопросы и решения
- Вендор reverse-image. Пороги/баланс ложных срабатываний. Блок vs мягкая задержка.

## 14. Критерии готовности (DoD)
- [ ] pHash считается при загрузке; дубликаты между листингами кластеризуются.
- [ ] Украденные/контрафакт — флаг в очередь модерации (не жёсткий блок честных).
- [ ] DSA takedown + повторные нарушители; retention.
- [ ] Тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

### DEV
- **pHash near-dup в Postgres «быстро» нереалистичен без механизма:** Hamming-distance на 64-бит pHash в чистом Postgres = seq-scan (медленно при росте). Хранить `media_hashes.phash bigint`; для near-duplicate — BK-tree/предвычисленные бакеты или вынести в индекс-сервис (обозначить roadmap-триггер по объёму). Уточнить в §6, иначе «быстро» нереализуемо.
- **Async-пайплайн:** `media/complete` ставит задачу → Edge Function/cron считает pHash и пишет `media_flags` (не блокировать ответ). Библиотека: sharp + perceptual hash.
- **Поля:** `media_hashes.algo`/`version` (алгоритм pHash может меняться).
- **Reverse-image вендор (§13)** не выбран — внешний блокер для stolen-photo ветки; pHash-дубликаты — свои силы (верно приоритизировано).
- **Тесты:** дубликат между листингами → флаг; сток-фото не ложно-срабатывает (порог); reverse-image мок.

### SEC
- **S-11 — rate-limit на media-роуты (A-4, подтверждён):** `media/sign`+`complete` (per-user), `public` (per-IP) — pHash вешается на `/api/media/complete`, без лимита pHash-вычисление + спам upload = DoS/cost-abuse усиливается. Включить до pHash-хука.
- **S-10 — image-processing safety (High, RCE/SSRF/bomb):** hard-лимиты (max bytes, max megapixels), таймаут, изоляция/sandbox; обрабатывать ТОЛЬКО объект из своего storage по серверному пути, **никогда по URL из запроса** (анти-SSRF); strip EXIF.
- **EXIF/гео-PII:** фото могут нести EXIF-геолокацию → утечка адреса; strip EXIF при обработке; retention `media_hashes/media_flags`.
- **Reverse-image вендор — GDPR:** отправка пользовательских фото третьей стороне → Art.28 DPA + минимизация (вписать в §8, F4).
- **RLS на `media_hashes/media_flags`** — модерация/админ через `app_metadata`-роль (S-6), не участники.
- **Signed-URL (S-8):** evidence/media — короткий per-request signed-URL только авторизованному; путь не публиковать.

**Вердикт:** 🔄 — механизм near-dup поиска в Postgres, async-пайплайн, rate-limit media (S-11), image-processing safety (S-10), DPA вендора. Приоритет P1/LATER — ок.
