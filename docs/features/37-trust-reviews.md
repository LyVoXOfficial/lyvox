# PRD: Доверие — trust score, бейджи, отзывы

> **Статус кода:** 🟢 ГОТОВО (в рамках скоупа PRD 37 / F14 / B1)
> **Категория:** Есть
> **Приоритет:** P0
> **Зависит от:** [[30-identity-account]] (верификация), [[10-escrow-safe-deal]] (рейтинг по завершённым сделкам), [[35-chat-antifraud]] (риск-сигналы), [[60-trust-gamification]]

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Доверие — главный драйвер лояльности в C2C. Рейтинг должен быть набором проверяемых сигналов, а не накручиваемыми звёздами.
- **Исход.** Identity-bound, портативный trust score из verified-identity + завершённых escrow-сделок + истории споров + метрик ответа; отзывы — только после реального взаимодействия.
- **KPI:** доля продавцов с trust-сигналами; корреляция trust↔конверсия; накрутки ≈0.

## 2. Контекст и текущее состояние (As-is)
- `trust_score` таблица + `lib/trust/deriveTrust.ts`, `public_trust_score_access`, `trust_score_cascade`; `VerificationBadge`, `components/trust` (TrustGateProvider — гейты auth/verified).
- **Отзывы:** `reviews` миграция, chat-gated `create_review` RPC + триггер рейтинга, `/api/reviews`, `LeaveReviewForm`, агрегат на `/user/[id]` (недавно дофикшен forgeable chat-gate).
- Слабость: trust_score питается частично (нет привязки к завершённым сделкам — их пока нет); badges_awarded есть.

## 3. Пользователи и сценарии
- **Покупатель:** «Видеть уровень доверия продавца и почему он такой».
- **Продавец:** «Зарабатывать репутацию реальными сделками, которую нельзя сбросить пере-регистрацией (verified-tier)».

## 4. UX и поведение (детально)
- **Trust-бейдж** на карточке/профиле + объяснение составляющих (верификация, завершённые сделки, споры, скорость ответа).
- **Отзывы:** оставить только после контакта/сделки; агрегатный рейтинг; ответ продавца (roadmap).
- **Бейджи:** verified, надёжный продавец и т.д. ([[60-trust-gamification]] — прогресс).
- Состояния: нет данных (новичок) / накапливается / устойчивый.

## 5. Логика и правила
- **Сигналы:** verified identity, завершённые escrow-сделки ([[10-escrow-safe-deal]]), споры/отмены, скорость ответа, риск-флаги чата ([[35-chat-antifraud]]).
- **Анти-накрутка:** рейтинг/репутация — только из проверяемых событий (завершённые сделки), не из «звёзд на пустом месте».
- Портативность: привязка к verified identity (`sub`), нельзя сбросить пере-регистрацией (только verified-tier).
- Краевые: новичок без сигналов, спорные отзывы, удалённый аккаунт (cascade).

## 6. Данные (Data model)
- `trust_score`, `reviews`, `badges_awarded`. (При появлении сделок) `seller_ratings` только из завершённых сделок. RLS: публичное чтение агрегатов; запись — через RPC/триггеры. Cascade при удалении.

## 7. API / интеграции
- `/api/reviews`, `/api/top-sellers`; deriveTrust как серверная логика; интеграция с escrow/chat-сигналами.

## 8. Безопасность, приватность, комплаенс
- Защита от форджинга отзывов (chat/deal-gated). Прозрачность составляющих score. GDPR (удаление/каскад).

## 9. Аналитика
- Распределение trust-score, влияние на конверсию, доля отзывов после сделок, попытки накрутки.

## 10. Доступность и i18n
- Доступные бейджи с текст-альтернативами; i18n `trust.*`, `reviews.*` (NL/FR/EN).

## 11. Тестирование
- Unit: deriveTrust, агрегат рейтинга. API: reviews gate (нельзя без контакта). e2e: оставить отзыв после чата; trust-бейдж отображается.

## 12. План внедрения (Rollout)

## 13. Открытые вопросы и решения

## Ревью-требования и sign-off (2026-06-28)

### DATA (БЛОКЕР — фундамент F14)
- **Формулы trust-score не существует.** `deriveTrust.ts` = только identity-уровни L0–L4; `trust_score.score` — голый `int`, инкрементится непрозрачной `trust_inc()` из модерации. DoD «объяснение составляющих» неисполним. **Зафиксировать явную взвешенную формулу** (не оставлять «веса — открытый вопрос §13»):
  ```
  trust_score = round(100 * sigmoid(
      w_id*identity_level_norm + w_deal*completed_deals_norm + w_rev*bayes_rating_norm
    + w_resp*response_speed_norm − w_disp*dispute_rate − w_risk*chat_risk_flags_norm))
  ```
  Стартовые веса (версионируемые, калибруемые): `w_id=0.35, w_deal=0.25, w_rev=0.20, w_resp=0.10, w_disp=0.07, w_risk=0.03`. **Пока escrow нет → `w_deal=0`**, ре-нормировать на доступные сигналы, пометить score «ранний».
- **Хранение компонент (объяснимость):** таблица `trust_score_components(user_id, component, raw_value, normalized, weight, contribution, computed_at)`; UI «почему такой score» читает отсюда; пересчёт батч/триггер, не on-the-fly.
- **Reviews — анти-накрутка (два пробела):**
  - **Стэкинг по продавцу:** `unique(advert_id, reviewer_id)` НЕ мешает накидать рейтинг через N объявлений одного продавца → добавить `unique(subject_id, reviewer_id)` (или 1 отзыв на пару за rolling-90d).
  - **Chat-gate форджабелен** дешёвым контактом → снизить вклад отзывов от L0/L1 и свежих аккаунтов (`reviewer_trust_weight`, снапшот на момент отзыва).
  - Anomaly-метрика: ревью-velocity по продавцу → флаг в [[38-moderation-reports]].
- **`profiles.rating` = голый avg, default 5.0** (новичок «идеальный 5.0») → **байесовское сглаживание** `bayes = (C*m + Σr)/(C+n)` (m≈4.2, C≈5), вес по доверию ревьюера; default `NULL` для «нет отзывов» (UI: «Нет отзывов» ≠ «5.0»).
- **Data-leakage:** `top_sellers.avg_views` тянет из недедупленного `advert_views` (F11) → trust накручиваем фейковыми просмотрами. Закрыть дедупом + анти-leakage реестром (исключать `removed/fraud_flagged/blocked`).
- **Портативность по `sub`, не `user_id`:** `identity_reputation(identity_sub text pk, score, ...)`; пере-регистрация с тем же itsme не сбрасывает (связь с [[30-identity-account]] F10).

### DEV (фундамент F14)
- **Интерфейс сигналов `TrustSignal[]`** с источниками (verification, review, deal, chat_risk, dispute) — чтобы deal-сигнал подключился позже без рефактора; `trust_weights` конфиг + `trust_score.version`.
- Правило каскада отзывов при удалении аккаунта (анонимизация автора).
- Тесты: `deriveTrust` детерминирован при заданных сигналах; review-gate (нельзя без контакта).

### UX (фундамент F14)
- **Анти-социальное доказательство:** публичные счётчики (отзывы/рейтинг) показывать только при пороге; новичок без отзывов — «Нет отзывов» + качественные сигналы (verified), не «5.0».
- Бейджи с текст-альтернативами (a11y); объяснение составляющих доступно с клавиатуры/скринридера.

**Вердикт:** ⛔ Блокер по DATA — нет формулы и хранимых компонент score, reviews-стэкинг, avg без байеса, leakage из `advert_views`. После явной формулы + `trust_score_components` + анти-стэкинг + байес + дедуп → 🔄/✅.
