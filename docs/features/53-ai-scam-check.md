# PRD: AI «это скам?» — проверка чата/ссылок

> **Статус кода:** ⛔ НЕ НАЧАТО (есть regex-скрабинг чата, AI-оценки нет)
> **Категория:** Бонус
> **Приоритет:** P2
> **Зависит от:** [[35-chat-antifraud]], [[38-moderation-reports]], [[10-escrow-safe-deal]] (реальный «замок»)
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Скрабинг по regex обходится; пользователю нужна понятная оценка «это похоже на скам».
- **Исход.** AI-классификатор диалога/ссылки даёт пользователю предупреждение и кормит риск-скор; усиливает (не заменяет) escrow и скрабинг.
- **KPI:** доля помеченных скам-диалогов; снижение успешных уводов; ложные срабатывания.

## 2. Контекст и текущее состояние (As-is)
- `scrubContacts` (regex) есть; AI-оценки рисков диалога — нет.

## 3. Пользователи и сценарии
- **Покупатель/продавец:** «Получить предупреждение, если собеседник ведёт по скам-паттерну».
- **Модерация:** «Получить AI-риск как сигнал».

## 4. UX и поведение (детально)
- Инлайн-предупреждение при высоком риске: «Похоже на мошенничество: вас просят уйти/оплатить по ссылке».
- (Опц.) кнопка «Проверить на скам» для ссылки/сообщения.
- Состояния: ок/подозрительно/высокий риск.

## 5. Логика и правила
- AI-классификация паттернов (увод, фейк-курьер, «оплати-чтобы-получить»); риск-скор → trust/модерация ([[35-chat-antifraud]]/[[38-moderation-reports]]).
- **Рамка:** деттеррент+сигнал, не стена; реальный контроль — escrow.
- Краевые: ложные срабатывания, многоязычность (NL/FR/EN/RU).

## 6. Данные (Data model)
- `message_risk_flags` (общая с чатом). Лог решений. Приватность сообщений.

## 7. API / интеграции
- Edge Function/LLM на send-хуке (асинхронно), интеграция с риск-скором.

## 8. Безопасность, приватность, комплаенс
- Обработка сообщений — минимально; прозрачность; не оверселлить.

## 9. Аналитика
- Доля помеченных, действия пользователя на предупреждение, ложные срабатывания.

## 10. Доступность и i18n
- i18n предупреждений (NL/FR/EN/RU).

## 11. Тестирование
- Unit/eval: классификатор на размеченных примерах. e2e: скам-паттерн → предупреждение.

## 12. План внедрения (Rollout)
- **LATER:** после базового анти-фрода. Усилие: M.

## 13. Открытые вопросы и решения
- Порог предупреждения. On-send vs on-demand. Вендор/модель.

## 14. Критерии готовности (DoD)

- [ ] Многоязычно; ложные срабатывания под контролем.
- [ ] Тесты/eval §11 ок.

## Ревью-требования и sign-off (2026-06-28)

Ревью: критичных замечаний именно по этому PRD немного; применить сквозные F-требования где релевантно.

### SEC / DATA
- **Опирается на `message_risk_flags` из PRD 35 (F14-смежно).** Эта таблица и агрегация сигнала в trust-score сейчас roadmap — поднять приоритет, т.к. данный PRD на неё ссылается. Формат сигнала: типы флагов + агрегация по conversation/пользователю для авто-mute.
- **Реальный контроль — escrow, не AI-предупреждение** (рамка в §5 верна; сохранить). `scrubContacts` усилить как **сигнал** (NFKC-нормализация, слитный BE-номер `^0\d{9}$`), но честно зафиксировать: deterrent, не контроль.
- **Обработка сообщений через внешний LLM — GDPR (F4):** DPA с вендором, минимизация передаваемого, прозрачность; обработка только server-side по доверенному пути (анти-SSRF, см. PRD 17 image-safety паттерн). Сообщения чата — в DSAR-реестр (F4/S-9).
- **`session`/серверный IP (F8)** для любых rate-limit вокруг on-demand «Проверить на скам».
- **Многоязычность eval** обязательна (NL/FR/EN/RU) — ложные срабатывания под контролем на каждом языке.

**Вердикт:** 🔄 — после фиксации зависимости от `message_risk_flags`/агрегации (PRD 35), DPA вендора LLM и DSAR-регистрации сообщений → ✅; самостоятельных блокеров нет.
