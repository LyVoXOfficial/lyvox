# LyVoX — UX и психологический аудит

Дата: 2026-06-27 · Аудитор: старший UX/продуктовый дизайнер
Источники: код репозитория (`apps/web/src`), live-наблюдения (`_live-capture-notes.md`), видение (`PROJECT_VISION_AND_TZ.md`), стратегия (`COMPETITIVE_STRATEGY.md`), макет (`design-preview.html`).
Позиционирование: **trust-first** барахолка для Бельгии (NL/FR/EN-first). Конкуренты: 2dehands, Facebook Marketplace, Vinted.

Метод для мобайла: live-скриншот снять нельзя, поэтому адаптив оценён по Tailwind-классам (`md:`/`lg:`/`hidden`), `bottom-nav`, `AdvertMobileContactBar`, `viewport-bottom-spacer` и логике `isMobile` в коде.

Легенда приоритетов: **High** — бьёт по конверсии/доверию сейчас; **Med** — заметное трение; **Low** — полировка.

---

## Сквозная архитектура чрома (layout)

`app/layout.tsx` рендерит в порядке: `TopBar` → `MainHeader` → `main` → `LegalFooter` → `ViewportBottomSpacer` → `BottomNav`. На мобайле трастовый `TopBar` скрыт (`hidden md:block`), `BottomNav` показан (`md:hidden`). Глобальный `<main>` имеет `pb-[calc(64px+safe-area)]`, что корректно резервирует место под нижнюю навигацию.

## 1. Home (`app/page.tsx` + `components/home/*`)

### Иерархия и раскладка
- Hero — двухколоночная сетка `md:grid-cols-[1.55fr_.92fr]`: слева eyebrow + H1 (с градиентной trust-фразой) + подзаголовок + крупный `SearchBar variant="hero"` + 4 чипа категорий; справа — карточка-статистика + CTA «Подать объявление». H1 один (хорошо).
- Ниже фолда по порядку: ссылка-блок в Discover → `InfoCarousel` → категории (`CategoriesCarousel`) → «Рекомендуем сейчас» (`DiscoveryFeed`, последние 24) → `TopSellersCarousel` → `TopAdvertCard` → «Бесплатные объявления» → «Недавно просмотренные».
- **Проблема: на десктопе первый экран почти не содержит товара.** Витрина (`DiscoveryFeed`) живёт глубоко под фолдом, после двух каруселей. Для маркетплейса «увидел товар → захотел» это слабый первый экран: hero продаёт идею, но не показывает ассортимент.

### Психология и доверие
- Сильный trust-месседж в H1 + подзаголовок; градиентный «trust phrase» — хороший акцент. Чип «Проверенные продавцы» выделен `lyvox-trust-gradient` — правильный приоритет.
- **Анти-социальное доказательство (критично для trust-first).** Карточка статистики (`page.tsx` ~408-501) показывает `latestAds.length` / `verifiedCount` / `freeAds.length`. На пустом маркете это буквально «4 / 4 / 1». Малые числа на видном месте подсознательно сигналят «тут пусто/никого нет» — для барахолки это худший из возможных первых сигналов (ликвидность — вопрос жизни и смерти, см. стратегию). Сейчас это не социальное доказательство, а его инверсия.
- `TopSellersCarousel` и `TopAdvertCard` корректно возвращают `null` при пустых данных (хорошо — не плодят пустых секций), но `latestAds.length` в статистике скрыть нельзя — оно всегда на экране.
- Нет ни одного отзыва/цитаты/«как это работает в 3 шага» на главной — для trust-first витрины это пробел: новичку не объясняется, *почему* тут безопаснее, чем на 2dehands.

### Трение и когнитивная нагрузка
- **Дубль поиска.** `MainHeader` рендерит `SearchBar variant="header"` (sticky, всегда виден), и hero рендерит `SearchBar variant="hero"`. На первом экране два одинаковых по функции поля поиска в ~80px друг от друга — расфокус внимания и сомнение «какое правильное».
- «Загрузка категорий…» — `CategoriesCarousel` и автокомплит `SearchBar` тянут categories клиентом на каждой странице (нет кэша схемы) → перцептивная медлительность.

### Мобайл vs десктоп
- Карточка-статистика `hidden ... md:flex` — на мобайле её нет (это, как ни странно, спасает от анти-соц-доказательства на мобайле). Но и CTA «Подать объявление» из неё на мобайле теряется — остаётся только `+` в bottom-nav.
- Hero-сетка схлопывается в одну колонку (ок). H1 `clamp(2rem,5vw,46px)` адаптивен.

### Чего не хватает
- Витрины на первом экране (хотя бы строки «горящих» карточек выше каруселей).
- Блока «как это работает» / «почему безопасно» с конкретными trust-механизмами.
- Честного социального доказательства вместо счётчиков на пустом складе.

### Рекомендации
| Приоритет | Рекомендация | Где |
|---|---|---|
| High | Убрать дубль поиска: либо скрыть header-поиск на home (`hidden` при `pathname === "/"`), либо убрать hero-поиск. | `main-header.tsx`, `page.tsx:371` |
| High | Заменить «голые» счётчики `4/4/1` на пороговую/качественную подачу: при `latestAds.length < N` показывать ценностные тезисы («Проверенные продавцы», «Без комиссии», «Чат внутри платформы») вместо цифр. Числа показывать только когда они работают на доверие. | `page.tsx:408-501` |
| High | Поднять витрину товаров выше фолда (строка карточек сразу после hero, до каруселей). | `page.tsx:505-559` |
| Med | Добавить блок «3 шага безопасной сделки» / отзывы на главной. | новый блок |
| Med | Кэшировать схему категорий (один фетч, общий стор) — убрать «Загрузка категорий…». | `categories-carousel`, `SearchBar.tsx:60` |

## 2. Discover / свайп (`app/discover/page.tsx` + `components/discover/SwipeDeck.tsx`, `SwipeCard.tsx`)

### Иерархия и раскладка
- Центрированный заголовок «Свайп-режим» + подзаголовок; чипы Drops (Только что / До €50 / Проверенные / Дешевле) + «Сбросить предпочтения»; колода `aspect-[3/4]`; внизу кнопки pass/act/like. Чисто и сфокусированно.
- Карточка (`SwipeCard`) намеренно **не показывает имя продавца** (identity gating) — соответствует стратегии. Показывает фото, тайтл, цену, локацию, лайки, сердечко.

### Психология и доверие
- Жесты + кнопки + клавиатура (стрелки) — отличная доступность.
- `act` (свайп вверх) открывает sheet: «Открыть объявление / Написать / Сделать оффер» с `requireTrust("verified", …)` — грамотная воронка к верифицированному контакту.
- На карточке нет trust-бейджа продавца (есть только `sellerVerified` в данных, но визуально не выводится) — упущенный trust-сигнал прямо в момент решения.

### Трение и пустые состояния (приоритетная зона)
- **Пустой первый экран.** `discover/page.tsx` передаёт `<SwipeDeck initial={[]} … />`, `page=0`. На маунте: `queue=[]`, `top=null`, `loading=false` → секунду показывается **empty-state** до того, как `loadMore()` подтянет первую страницу. То есть пользователь при входе видит «Вы всё просмотрели» раньше, чем приедут карточки. Это ложный empty-state на старте.
- **Empty-state существует** (`SwipeDeck.tsx:279-296`, ключи `discover.empty_title = "You're all caught up"`). Вопреки live-заметке («нет empty-state») — он есть. Реальная проблема в другом: на крошечном каталоге `filterUnseen()` (localStorage `seenAdverts`) быстро вычищает все 3-4 карточки, и пользователь получает «Вы всё просмотрели» — что на полупустом маркете читается как «тут мёртво».
- Текст empty-state «You're all caught up / No more listings» нейтрален для богатого каталога, но **вреден для пустого**: не зовёт назад в поток с пользой.
- Кнопка empty-state ведёт «Back to the feed» (`/`) — разумно, но не предлагает «Сбросить просмотренные» (хотя `resetTaste()` сбрасывает только вкусы, не `seenAdverts`).

### Мобайл vs десктоп
- `max-w-md`, `aspect-[3/4]`, touch-жесты (`touch-none`, `setPointerCapture`) — это в первую очередь мобильный паттерн, на десктопе работает через кнопки/клавиатуру. Адаптив корректный.

### Чего не хватает
- Защиты от ложного empty-state на старте (различать «ещё не грузили» и «реально пусто»).
- Сброса `seenAdverts` при исчерпании колоды на малом каталоге.
- Trust-бейджа на карточке.

### Рекомендации
| Приоритет | Рекомендация | Где |
|---|---|---|
| High | Не показывать empty-state, пока не было ни одной загрузки. Ввести флаг `hasLoadedOnce`; `isEmpty = hasLoadedOnce && !top && !loading`. | `SwipeDeck.tsx:237` |
| High | При исчерпании колоды на малом каталоге предлагать «Показать снова» со сбросом `seenAdverts` (а не только taste). | `SwipeDeck.tsx:279-296`, `lib/seenAdverts` |
| Med | Передавать SSR-сид первой страницы (`initial`) из `discover/page.tsx`, чтобы первый кадр был с карточками, а не пустой. | `discover/page.tsx:19` |
| Med | Вывести trust-бейдж (`sellerVerified`) на `SwipeCard`. | `SwipeCard.tsx` |
| Low | Переписать текст empty-state дружелюбнее + кнопка «Сохранить поиск, чтобы узнать о новинках». | i18n `discover.empty_*` |

## 3. Search (`app/search/page.tsx` + `components/SearchFilters.tsx`, `SearchBar.tsx`)

### Иерархия и раскладка
- Шапка результатов (карточка): H1 «{query} в Бельгии» + счётчик «Показано N–M из T» + сортировка + «Сохранить поиск» (`SaveSearchButton`) + ссылка в Discover + (мобайл) кнопка фильтров. Ниже — чипы активных фильтров. Затем 2 колонки: sticky-рейл фильтров (`lg:block`, 262px) + грид результатов (`grid-cols-2 lg:grid-cols-3`).
- Каркас сильный и профессиональный; счётчик результатов **есть** (вопреки live-заметке) — `search.showing` с start/end/total.

### Психология и доверие
- Тумблер «Только верифицированные» в фильтрах — прямой trust-фильтр, правильно вынесен.
- «Сохранить поиск» присутствует — хороший retention-крючок (стратегия называет saved searches «must-have»).
- Пустое состояние (`AdsGrid` emptyTitle/emptyDescription/primaryAction) сделано хорошо: разные тексты для «с фильтрами» и «без», кнопки «Смотреть категории» / «Очистить фильтры». Это сильное место.

### Трение и когнитивная нагрузка (приоритетная зона)
- **Медленный first paint результатов.** При загрузке (`loading`) рендерится одиночный спиннер `Loader2` в карточке (`page.tsx:615-627`), а не скелетон-грид. При этом `AdsGridSkeleton` импортирован (`page.tsx:12`), но **не используется** — готовый скелетон есть, но не подключён. Спиннер вместо скелетона усиливает ощущение медлительности.
- Зависимость результатов от загрузки каталога в фильтрах: «Загрузка…»/«Categories unavailable» в рейле, пока тянется `categories`.
- Фильтры применяются мгновенно при каждом изменении (`applyFilters` → `router.push`) — это и плюс (живо), и минус: каждое касание слайдера/чекбокса = новый запрос и перерисовка. Кнопка «Применить» при этом тоже есть — двойственная модель (мгновенно + кнопка) сбивает с толку.

### Мобайл vs десктоп
- Десктоп: sticky-рейл слева. Мобайл: фильтры в Sheet-драуэре (триггер-пилюля с счётчиком активных фильтров) — корректный паттерн.
- **Инфинит-скролл на мобайле** реализован вручную через scroll-listener + `router.push(page+1)` с `setTimeout(2000)` сбросом флага (`page.tsx:328-357`). Это хрупко: фиксированная задержка 2с не привязана к фактическому завершению фетча → возможны дабл-загрузки или подвисания. Десктоп использует нумерованную пагинацию — две разные модели.

### Чего не хватает
- Скелетон-грида вместо спиннера.
- Единой модели применения фильтров (или мгновенно, или по кнопке).
- На мобайле — отсутствие trust-полосы (TopBar скрыт) лишает результаты trust-контекста.

### Рекомендации
| Приоритет | Рекомендация | Где |
|---|---|---|
| High | Подключить `AdsGridSkeleton` вместо спиннера в loading-ветке. | `page.tsx:615-627` (импорт уже есть, стр. 12) |
| Med | Убрать двойственность: либо мгновенные фильтры без кнопки «Применить», либо кнопка-применение. | `SearchFilters.tsx:1170-1179` |
| Med | Заменить ручной scroll-listener инфинит-скролла на `IntersectionObserver` со sentinel и флагом по факту завершения фетча. | `page.tsx:328-357` |
| Low | Кэшировать схему категорий (общая с home). | `SearchFilters.tsx:250-292` |

## 4. Ad detail (`app/ad/[id]/page.tsx` + `AdvertContactPanel`, `AdvertMobileContactBar`, `SellerIdentityGate`)

### Иерархия и раскладка
- 2 колонки `lg:grid-cols-[minmax(0,1fr)_372px]`: слева галерея → пилюля состояния/локация/дата → H1 + цена + лайк → описание → карточка продавца / `SellerIdentityGate` → блок «Доверие и безопасность» → спецификации/поколение/инсайты (для авто); справа sticky `AdvertContactPanel` (+ `TraderPanel` для бизнеса). Внизу — форма отзыва (если есть право) + похожие.
- Это **самая сильная страница продукта**. Иерархия логична, trust-блоки на месте.

### Психология и доверие (эталон)
- **Identity gating**: `canSeeSeller = viewerVerified || isOwnListing`. Невериф. видит `SellerIdentityGate` («Подтвердите, чтобы увидеть продавца») — точное попадание в trust-стратегию.
- `AdvertContactPanel`: бейджи «Проверен»/«Телефон подтверждён»/«Checks pending», CTA «Написать продавцу» через `requireTrust("verified")`, safety-tip «LyVoX никогда не обрабатывает платежи», `ReportButton`. Анти-скам формулировки («избегайте внешних платёжных ссылок») — грамотно.
- Блок «Доверие и безопасность» — таймлайн из 4 проверок (email/телефон/чат/«осмотр до оплаты»). Снижает тревогу сделки — именно то, что нужно trust-first.
- JSON-LD Product/Car + BreadcrumbList — хорошо для SEO.

### Трение
- Описание тестовое («test test») — это контент, не код; но напоминание: на пустом каталоге карточка выглядит «ненастоящей».
- «Местоположение не указано» (`location_not_specified`) — на барахолке локация критична для доверия и логистики; стоит делать её обязательной при создании.
- `revalidate = 0` + `force-dynamic` + множество последовательных запросов в `loadAdvertData` (advert → category → breadcrumbs → specifics → media → make/model/color/generations/insights/options → profile → trust → count → benefits → business) — длинная серия `await` подряд. Это может давать заметный TTFB на карточке. Часть запросов можно распараллелить (`Promise.all`).

### Мобайл vs десктоп
- `AdvertMobileContactBar` (`lg:hidden`) — sticky-бар внизу: цена + «Написать продавцу», та же `requireTrust/startChat` логика. Позиционируется `bottom: 76px` (над bottom-nav высотой 76px) — стыкуется корректно. Страница даёт `pb-[160px] lg:pb-0` под оба бара. **Адаптив здесь сделан правильно.**
- Sticky правый рейл (десктоп) → нижний бар (мобайл) — образцовый паттерн.

### Чего не хватает
- Trust-score продавца показан числом без объяснения, что оно значит (на карточке продавца есть `trustScoreLabel`, но без шкалы/контекста «что такое 0 / 50 / 100»).
- Нет «безопасной сделки»/escrow CTA (это roadmap, но место под него тут).

### Рекомендации
| Приоритет | Рекомендация | Где |
|---|---|---|
| Med | Распараллелить независимые запросы в `loadAdvertData` (`Promise.all` для category/specifics/media/profile/trust/benefits) — снизить TTFB. | `ad/[id]/page.tsx:1259-1614` |
| Med | Сделать локацию обязательной при создании; на карточке не показывать «не указано» как норму. | `post/PostForm`, `page.tsx:378` |
| Low | Добавить пояснение к trust-score (тултип «из чего складывается»). | `SellerCard.tsx` |
| Low | Зарезервировать место/плейсхолдер под будущий «Safe deal» CTA в контакт-панели. | `AdvertContactPanel.tsx` |

## 5. Post / создание объявления (`app/post/page.tsx` + `PostForm.tsx`)

### Иерархия и раскладка
- Гейтинг до формы: не залогинен → карточка «Войдите»; залогинен, но не верифицирован → жёлтая карточка «Подтвердите аккаунт» (email/телефон) с кнопками «К верификации» / «Профиль». Только верифицированный видит `PostForm` — 8-шаговый визард (`TOTAL_STEPS = 8`) с автосейвом (`AUTO_SAVE_INTERVAL_MS = 30_000`).
- Логика гейтинга trust-корректна (только вериф. публикуют) и хорошо объяснена («подтверждённые контакты снижают мошенничество»).

### Психология и доверие
- Объяснение «зачем верификация» прямо в гейте — снижает фрустрацию от барьера. Хорошо.
- Автосейв + черновики — снижают страх потери данных в длинной форме.

### Трение и когнитивная нагрузка (риск)
- **8 шагов — это много для C2C-барахолки.** Стратегия прямо требует «sub-60s listing, photo-first» как single-player-утилиту для привлечения продавцов до появления покупателей. 8-шаговый визард противоречит этому: для трактора/недвижимости оправдано, для «продать диван» — избыточно. Высокий риск отвала на середине.
- Двойной барьер на входе: сначала логин, потом обязательная верификация **до** того, как человек вообще увидел форму. Для первого опыта это может отпугнуть («ещё ничего не создал, а уже два барьера»). Альтернатива: дать заполнить черновик, верификацию требовать на шаге публикации.
- (Из стратегии/бэклога) `prompt()` для OTP доп. телефона — нативный браузерный prompt вместо модалки.

### Мобайл vs десктоп
- `container mx-auto max-w-3xl p-4` — адаптивно. Но 8 шагов на мобайле = много вертикального скролла/тапов; критично, что именно мобайл — основной канал (стратегия mobile-first).

### Чего не хватает
- Быстрого пути «фото → черновик за 60 секунд» (AI-ассист из стратегии — позже, но упрощённый flow можно сейчас).
- Прогресс-индикатора ценности («ещё 2 шага до публикации»).

### Рекомендации
| Приоритет | Рекомендация | Где |
|---|---|---|
| High | Разрешить заполнять/сохранять черновик до верификации; жёсткий гейт верификации — только на финальном шаге публикации. | `post/page.tsx:126-174` |

### Психология и доверие

## 9. Auth / Onboarding (`login`, `register`, `onboarding`)

### Login (`login/page.tsx`)
- Табы «Пароль» / «Magic link», соц-логин (Google/Facebook), «запомнить устройство», валидация email через `loginSchema`, тосты ошибок. Зрело.
- **Хардкод-строка**: `'Password must contain at least 8 characters.'` обёрнута в `tr(..., fallback)`, но ключ `auth.errorPasswordMinLength` должен существовать во всех локалях — иначе бельгийский пользователь увидит англ. фолбэк. (Сверить наличие ключа в nl/fr.)
- После входа редирект на `next ?? "/profile"` — разумно.

### Register (`register/page.tsx` + `RegisterForm`, `messages.ts`)
- Локаль определяется по `?lang`/Accept-Language на сервере (хорошо для BE). Но (из стратегии quick-win) `register/messages.ts` маппит `ru: englishMessages` — русскоязычный пользователь на регистрации видит **английский**, хотя `ru.json` полон. Для русско/украиноязычной диаспоры (заявленный вторичный сегмент) это баг доверия на критическом шаге.

### Onboarding (`onboarding/page.tsx`)
- Чеклист готовности аккаунта (подтвердить email / профиль / телефон) + «почему это важно» + «можно уже смотреть маркет». Хороший progressive-onboarding: не блокирует, объясняет ценность каждого шага.
- **Хардкод английского**: `"Account readiness"`, `"This step improves account trust and reduces support checks later."`, заголовок `"Why this matters"`, `"Browse marketplace"` — зашиты строками, не идут через локаль `t`. На nl/fr/ru эти куски останутся английскими.
- Локализованный контент (`messages[locale]`) для заголовка/чеклиста есть, но `ru` присутствует — а вот окружающая «обвязка» (3 строки) — английская для всех.

### Психология и доверие (общее по auth)
- Соц-логин снижает трение регистрации (хорошо).
- Trust-нарратив проходит через onboarding (объяснение пользы верификации) — согласованно с позиционированием.

### Мобайл vs десктоп
- `register`: `max-w-5xl`, `onboarding`: `md:grid-cols-[1fr_360px]` (на мобайле сайдбар «почему важно» уходит вниз) — адаптивно.

## Сквозные проблемы (через все страницы)

### A. Мерцание авторизации в шапке (High)
`UserMenu.tsx` и `main-header.tsx` стартуют с `email=null`/`hasSession=null` и определяют сессию клиентом (`supabase.auth.getUser()` + `onAuthStateChange`) уже после гидрации. Поэтому **залогиненный пользователь на доли секунды видит гостевые «Войти/Регистрация»**, после чего UI переключается на аккаунт (и наоборот при выходе). Это классический trust-шум: «сайт меня не помнит». Корень — отсутствие серверного знания о сессии в шапке (SSR рендерит гостевой вид).
**Фикс**: пробрасывать начальное состояние сессии с сервера (через layout/контекст или серверный компонент-обёртку шапки), либо рендерить нейтральный скелетон аватара вместо гостевых кнопок до резолва. Файлы: `main-header.tsx:37-72`, `UserMenu.tsx:42-136`.

### B. Анти-социальное доказательство на пустом маркете (High)
Счётчики `4/4/1` на home (раздел 1) — самый опасный паттерн для trust-first барахолки в фазе cold-start. Малые числа кричат «пусто». Нужно прятать/качественно заменять, пока ликвидность не наберётся.

### C. Дубль поиска на home (High)
Header-поиск (sticky) + hero-поиск одновременно на первом экране (раздел 1).

### D. Перцептивная медлительность от клиентских фетчей каталога (Med)
`categories` тянутся клиентом в `SearchBar`, `SearchFilters`, `CategoriesCarousel`, форме поста — каждый со своим «Загрузка…», без общего кэша. Плюс спиннер вместо скелетона на search. Кэшировать схему один раз.

### E. i18n-протечки и RU-баг (High/Med)
- `ru → englishMessages` в register/onboarding (русскоязычная диаспора — заявленный сегмент).
- Хардкод-строки onboarding.
- Дефолт-локаль для BE: middleware `defaultLocale=en` + cookie `ru` в тест-браузере; для Бельгии корректнее гео/Accept-Language → nl/fr.
- Риск англ.-фолбэков, где ключей нет в nl/fr.

### F. Потеря trust-контекста на мобайле (Med)
`TopBar` с трастовыми тезисами — `hidden md:block`. На мобайле (основной канал!) трастовая полоса полностью исчезает. Стоит дать компактный мобильный аналог (например, одна сворачиваемая строка-сигнал или вынести в hero).

### G. Заглушки, выглядящие рабочими (Med)
Reviews (`reviews: []`), статичные «Enabled» уведомления в профиле, вечно пустой таб отзывов. Для trust-first честность важнее видимости фич: лучше скрыть, чем имитировать.

### H. Две разные модели пагинации/применения (Med)
Search: нумерованная пагинация (десктоп) vs ручной инфинит-скролл (мобайл, хрупкий setTimeout). Фильтры: мгновенно + кнопка «Применить» одновременно.

### I. Неиспользуемый/двойной код чрома (Low)
`components/footer.tsx` не используется (в layout — `LegalFooter`). `AdsGridSkeleton` импортирован в search, но не подключён. Технический долг, влияющий на UX (скелетон не показан).

### J. Глубина воронки продавца (High для роста)
8 шагов + двойной барьер (логин → верификация до формы) против стратегической цели «листинг за 60 секунд». Прямо тормозит набор supply в cold-start.
