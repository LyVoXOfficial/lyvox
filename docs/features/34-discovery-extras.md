## 1. Зачем это (Problem & Outcome)
- **Проблема.** Удержание и повторные визиты — слабое место C2C. Нужны инструменты, возвращающие пользователя (сохранённые поиски + алерты — топ-рычаг ретеншена).
- **Исход.** Сохранённые поиски с алертами, рекомендации/«похожие», сравнение, недавно просмотренное, избранное и лайки — единая дискавери-экосистема.
- **KPI:** доля с ≥1 сохранённым поиском; CTR алертов; возвраты D7; использование сравнения/избранного.

## 2. Контекст и текущее состояние (As-is)
- **Saved searches + алерты:** `saved_searches`, `saved_search_alerts`, `/api/saved-searches`, cron `/api/cron/saved-search-alerts`, `lib/savedSearches.ts`, экран `/saved`.
- **Рекомендации/вкус:** `lib/taste.ts` (localStorage), `lib/popularity.ts`, `lib/discoveryFeed.ts`, `SimilarAdverts`, `RecentlyViewed(+Recorder)`, `lib/recentlyViewed.ts`, `lib/recentSearches.ts`, `lib/seenAdverts.ts`.
- **Сравнение:** `lib/comparison/{index,scoring}`, `components/comparison`, `/api/comparison`.
- **Избранное/лайки:** `favorites` + `advert_likes` (раздельно), `/api/favorites`, `/api/likes`, `FavoriteToggle`, `LikesProvider`.
- Нет: price-drop алертов ([[56-price-drop-alerts]]); серверной кросс-девайс персонализации.

## 3. Пользователи и сценарии
- «Сохранить поиск и получать новые по нему»; «сравнить 2–3 товара»; «вернуться к недавно смотренному»; «лайкать/в избранное».

## 4. UX и поведение (детально)
- **Saved search:** из выдачи «сохранить поиск» → управление в `/saved` → алерт (in-app/email/пуш [[36-notifications]]).
- **Рекомендации:** «похожие» на карточке; персональная лента по taste (cold start = популярное/новое).
- **Сравнение:** добавить в сравнение → таблица атрибутов + scoring.
- **Избранное vs лайки:** избранное — личный список; лайк — публичный сигнал/счётчик. Разъяснить разницу в UI.
- Состояния: пусто/есть/ошибка для каждого модуля.

## 5. Логика и правила
- Taste: веса like+1/fav+2/pass−1 (+ down−2 из [[01-discover-swipe-system]]); rerank; уважение consent (`functional`).
- Saved-search алерты: дедуп, частота, отписка/quiet hours.
- Сравнение: ограничение числа позиций; scoring-формула.
- Краевые: устаревшие сохранённые поиски, удалённые объявления в избранном.

## 6. Данные (Data model)
- `saved_searches`, `saved_search_alerts`, `favorites`, `advert_likes`. Taste/recent — localStorage (приватность). (Опц.) серверная персонализация — roadmap.
- RLS owner-only; экспорт/удаление (GDPR).

## 8. Безопасность, приватность, комплаенс

## 9. Аналитика

## 10. Доступность и i18n

## Ревью-требования и sign-off (2026-06-28)

### UX (фундамент F14 — анти-соц-доказательство + механики)
- **«Избранное» vs «лайки» — дать механику различия, не только «разъяснить»:** разные иконки/смысл — 🔖 «Сохранить» (личный список, приватно) vs 👍 (публичный сигнал продавцу, влияет на выдачу); микротекст при первом использовании каждого. Не два сердечка.
- **Анти-социальное доказательство:** публичный `likeCount` показывать только при пороге `>= N` (старт 3–5); ниже — не рендерить число (а не «0»).
- **UX-дефолты алертов (анти-усталость):** дефолт — дайджест (не мгновенно), max 1 уведомление/день на поиск, «Пауза на 30 дней» и «Удалить поиск» в один тап.
- **Лимит сравнения 2–4** (рекоменд. 3) + «Показать отличия» (подсветка разнящихся строк).
- **Краевые состояния:** проданное/удалённое в избранном — пометка «Снято с продажи» + «Похожие» (не молча исчезает); видимая «Очистить историю» для недавно просмотренного.
- **a11y:** сердечко/лайк — `role=button`+`aria-pressed`+`aria-label`; таблица сравнения — `<table>`/`<th scope>`; управление сохранёнными поисками — кнопки с `aria-label` (Пауза/Удалить/Изменить); тосты — `aria-live="polite"` без кражи фокуса; состояние пусто/ошибка текстом; тач-таргеты ≥44px.

### DEV (фундамент F11)
- **Дедуп saved-search алертов:** cron есть, но ключ дедупа не задан → дубли. Добавить `unique(saved_search_id, advert_id)` в `saved_search_alerts` (или курсор `last_seen_advert_id`/`last_run_at`); частота max 1 дайджест/сутки/поиск; quiet hours из [[36-notifications]].
- **Удалённые favorites:** правило отображения (tombstone «Снято с продажи», не молча скрывать).
- Тесты: cron не шлёт повтор; quiet hours отложил.

### DATA (фундаменты F6, F11, F14)
- **Серверный лог taste-сигналов** (даже при клиентском rerank): `taste_signals(user_id|anon_id, advert_id, signal, weight, ts)` под `functional`-consent — иначе рекомендации неаудируемы (precision@k, like-rate uplift A/B).
- **Decay в taste:** `weight * exp(-Δt/τ)`, τ ~ 30 дней; L2-нормализация вектора пользователя.
- **Cold-start на надёжном сигнале:** популярность считать из дедупленных просмотров + контактов (F11), не из сырых `advert_views` (накручиваемы).
- **Единое определение engagement:** документировать раздельный учёт `like` (публичный → taste) vs `favorite` (личный → seller-analytics).
- GDPR: экспорт/удаление аккаунта чистит `taste_signals`/события (F6).
