# PRD: Dispute-движок (споры по сделкам)

> **Статус кода:** ⛔ НЕ НАЧАТО (на уровне сделок; есть только модерация объявлений/жалобы)
> **Категория:** Не хватает (критическое)
> **Приоритет:** P0
> **Зависит от:** [[10-escrow-safe-deal]] (выкатывать вместе — без споров escrow нельзя запускать), [[12-shipping-integration]] (трекинг как доказательство), [[38-moderation-reports]]
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** В день, когда появляется escrow, появляются конфликты «не пришло / не то / повреждено». Без честного, человеческого, быстрого разбора «trust-first» бренд рушится (слабая поддержка измеримо *разрушает* доверие). Непрозрачные решения — главная жалоба у всех конкурентов.
- **Исход.** Обе стороны видят **один и тот же таймлайн доказательств** (фото, трекинг, лог чата) и получают мотивированное решение с правом апелляции и SLA. DSA Art.20 (человеческий разбор) — как фича, не как мелкий шрифт.
- **KPI:** медиана времени до решения; доля апелляций; доля «замороженных» средств; CSAT по спорам.

## 2. Контекст и текущее состояние (As-is)
- Есть `reports` (жалобы на объявления, 5 причин, rate-limit, очередь), AI-moderation/fraud-rules — но **нет споров на уровне транзакции** с заморозкой средств.
- Нет таблиц `disputes/dispute_messages/dispute_evidence`.

## 3. Пользователи и сценарии
- **Покупатель:** «Товар не пришёл / не соответствует — хочу открыть спор и вернуть деньги».
- **Продавец:** «Хочу приложить доказательства отправки/состояния и защититься».
- **Модератор/Support:** «Хочу видеть единый таймлайн и вынести решение с обоснованием».
- Happy-path: покупатель в окне осмотра открывает спор → загружает фото/описание → продавец отвечает доказательствами → модератор решает (возврат/release/частично) → апелляция при несогласии.

## 4. UX и поведение (детально)
- **Открытие:** кнопка «Открыть спор» доступна в статусах оплачено/отправлено/доставлено в пределах окна. Причины: не получено / не соответствует / повреждено / другое.
- **Кабинет спора:** общий таймлайн (события сделки + сообщения + доказательства), таймеры ответов, статус.
- **Доказательства:** фото/видео при отправке и возврате; трекинг подтягивается из [[12-shipping-integration]].
- **Решение:** возврат покупателю / release продавцу / частичный сплит; обязательное текстовое обоснование.
- **Апелляция:** одна попытка, отдельная очередь, срок.
- **Состояния:** открыт → в ответе продавца → на решении → решён → апелляция → закрыт.
- **SLA-копирайт:** публичный срок реакции (бизнес-часы на старте).

## 5. Логика и правила
- **Заморозка средств** на время спора (через [[10-escrow-safe-deal]]); release заблокирован.
- **Конкаррентный лимит** открытых споров = пропускной способности одного человека; авто-троттлинг новых safe-deal при достижении потолка (защита соло-оператора).
- **Refund/return abuse:** никогда не возврат без подтверждения доказательств; для «возврат без возврата товара» — требовать подтверждение продавца/доказательства.
- **Таймеры:** дедлайн ответа продавца → дефолтное решение при молчании.
- **Краевые:** частичная доставка, потеря перевозчиком, «cambiazo»/подмена, повторные споры одного юзера (фрод-сигнал).

## 6. Данные (Data model)
- `disputes { id, deal_id, opened_by, reason, status, resolution, resolved_by, resolved_at, created_at }`
- `dispute_messages { id, dispute_id, sender_id, body, created_at }`
- `dispute_evidence { id, dispute_id, owner_id, media_path, kind, created_at }`
- `dispute_events { id, dispute_id, type, payload, created_at }` (иммутабельно)
- RLS: участники сделки + модератор/support; service-role server-only. Аудит решений.

## 7. API / интеграции
- `POST /api/disputes` (открыть), `POST /api/disputes/[id]/message`, `POST /api/disputes/[id]/evidence`, `POST /api/disputes/[id]/resolve` (роль модератор), `POST /api/disputes/[id]/appeal`.
- Интеграция с escrow: блок/release/refund средств; со shipping: тянуть трекинг.

## 8. Безопасность, приватность, комплаенс
- **DSA Art.20** — человеческий разбор + понятная апелляция. Аудит-лог всех решений.
- Доступ к доказательствам — только стороны + модератор; retention/удаление (GDPR).
- Анти-абуз: лимиты на открытие, фрод-скоринг повторных споров.

## 9. Аналитика
- `dispute_opened/responded/resolved/appealed`, время до решения, исход-распределение, доля апелляций, CSAT.

## 10. Доступность и i18n
- Таймлайн с ARIA; i18n `dispute.*` (причины, статусы, обоснования) — NL/FR/EN.

## 11. Тестирование
- Unit: таймеры/дефолтные решения, сплит-расчёт. API: открытие/ответ/решение/апелляция + RLS. e2e: спор → решение → возврат через escrow (test mode). Негатив: спор вне окна, доступ постороннего.

## 12. План внедрения (Rollout)
- Выкатывается **вместе с [[10-escrow-safe-deal]]** (NOW-2). Старт: бизнес-часы SLA, жёсткий потолок одновременных споров. Усилие: M–L.

## 13. Открытые вопросы и решения
- Окна ответов и SLA-цифры. Политика частичных сплитов. Порог авто-троттлинга. Кто решает споры при росте (найм support).

## 14. Критерии готовности (DoD)
- [ ] Спор замораживает release; решение двигает деньги корректно.
- [ ] Общий таймлайн доказательств виден обеим сторонам; решение мотивировано; есть апелляция.
- [ ] Конкаррентный лимит + троттлинг работают.
- [ ] Аудит-лог; RLS; GDPR-retention.
- [ ] Тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

### LEGAL
- **ADR/потребитель (добавить раздел):** для trader↔consumer споров — информировать о праве на BE Consumer Mediation Service (Service de médiation pour le consommateur / Consumentenombudsdienst), Dir 2013/11; решение LyVoX НЕ лишает потребителя CDE-средств (14 дней, conformity) и права на суд. Нигде не формулировать решение как «окончательно и не подлежит обжалованию».
- **Исправить §8 — ошибочная ссылка на DSA Art.20:** Art.20 касается модерации контента (зона PRD 38), а НЕ денежных споров по сделкам. Денежные споры регулируются договорным/потребительским правом + ADR. Убрать DSA Art.20 как основание здесь.
- **GDPR доказательств (§8):** правовое основание (Art.6(1)(b)/(f)); retention — до закрытия спора + срок апелляции + срок исковой давности по типу требования, затем удаление (не «навсегда»); минимизация EXIF/гео-метаданных фото/видео.
- **Art.22 GDPR — авто-дефолт:** «дефолтное решение при таймауте / авто-скоринг повторных споров не выносит окончательного денежного решения без возможности человеческого пересмотра по запросу стороны»; auto-default помечается и оспаривается.
- **Rulebook споров (внутренний документ, ссылка):** критерии исходов, equal-evidence (adversarial principle), сроки, запрет конфликта интересов — процедурная честность и защита от исков о произволе.

### DEV
- **S6 — атомарная заморозка release:** открытие спора и cron-release escrow — гонка. Open dispute → `deals.status='disputed'` атомарно; release заблокирован на уровне БД-constraint, не только в коде.
- **`resolve` — атомарность денег (S0/F1):** в одной логической операции записать `dispute_events`, перевести `deals.status`, вызвать провайдера (refund и/или transfer) под `idempotency_key=dispute:{id}:resolve`, проверить `payouts`/`refunded_amount`. Частичный сплит = две денежные операции, обе под идемпотентностью.
- **Поля (дополнить §6):** `disputes.respond_by timestamptz`, `resolution_amount_cents` (для сплита), `version int`. Таблица переходов состояний (S4).
- **Таймеры:** `POST /api/cron/dispute-timeouts` — дефолт-решение при истёкшем `respond_by`.
- **Роль support/модератора:** ввести `profiles.role in ('user','support','admin')` или `staff_roles`; RLS-функция `is_staff()` SECURITY DEFINER (по аналогии с `is_conversation_participant`). Сейчас роль выражена только через admin/`app_metadata`.
- **S3 — иммутабельность `dispute_events`** (insert-only + триггер).
- **Троттлинг:** счётчик открытых споров (view/функция) + флаг `disputes_intake_open`; `POST /api/deals` (PRD 10) проверяет его.
- **Тесты:** спор вне окна (403), доступ постороннего (RLS), сплит двигает обе суммы один раз, гонка resolve/release.

### SEC
- **Authorization роли модератора (High, S-6):** `resolve` (двигает деньги) сверяет роль из `app_metadata`/серверного источника, не из user-editable profiles-колонки; service-role на mutate + иммутабельный аудит решений (actor/reason/before-after).
- **IDOR на доказательства (High, S-8):** `dispute_evidence` — короткий per-request signed-URL только авторизованной стороне/модератору; путь объекта не публиковать; RLS через `is_deal_participant() OR is_moderator()`.
- **Заморозка release — race (High, S6):** атомарно в статус-машине сделки (БД-constraint, не код).
- **Refund/return abuse + brigading:** велосити-лимит открытия споров (per-user/per-IP с серверным IP — F8/S-1), фрод-скоринг повторных споров → trust-score.
- **Дефолт по таймауту:** баланс против абуза (покупатель-фрод провоцирует таймаут) + аудит авто-решений.

**Вердикт:** 🔄 — не блокер сам по себе (зависит от PRD 10), но обязательны: ADR-инфо, убрать ошибочный DSA Art.20, Art.22-гарантия авто-дефолта, retention доказательств, rulebook, атомарность resolve, роль support, RLS evidence.
