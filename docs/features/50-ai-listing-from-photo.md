# PRD: AI-листинг по фото

> **Статус кода:** ⛔ НЕ НАЧАТО
> **Категория:** Бонус
> **Приоритет:** P2
> **Зависит от:** [[31-listing-creation]], [[32-category-catalogs]], [[52-ai-fair-price]]
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Создание объявления — узкое место предложения (ликвидность). Ручное заполнение заголовка/категории/описания — трение.
- **Исход.** Из 1–3 фото AI предлагает заголовок, категорию, атрибуты, описание и цену → листинг за <60 сек. Пользователь редактирует и подтверждает (никогда не авто-публикация).
- **KPI:** время создания листинга; доля листингов через AI; рост числа новых листингов (supply).

## 2. Контекст и текущее состояние (As-is)
- Есть 8-шаговый мастер и категорийные схемы; AI-помощника по фото нет. Стратегия прямо называет это рычагом supply.

## 3. Пользователи и сценарии
- **Продавец:** «Сфоткал — получил готовый черновик, поправил, опубликовал».

## 4. UX и поведение (детально)
- Загрузка фото → «AI заполняет…» → предзаполненные поля (с пометкой «предложено AI», легко править) → подтверждение → обычный флоу публикации [[31-listing-creation]].
- Прозрачность: «черновик от AI, проверьте перед публикацией».
- Состояния: анализ/успех/низкая уверенность (просим уточнить)/ошибка (ручной ввод).

## 5. Логика и правила
- Vision-модель → категория/атрибуты/описание; цена — из [[52-ai-fair-price]].
- **Никогда не публиковать без подтверждения пользователя.**
- Краевые: несколько товаров на фото, плохое качество, незнакомая категория.

## 6. Данные (Data model)
- Без новых таблиц (черновик — существующий draft). (Опц.) лог AI-предложений для оценки качества. Приватность фото.

## 7. API / интеграции
- Edge Function/внешний vision-LLM; хук в мастер создания. Лимиты/стоимость вызовов.

## 8. Безопасность, приватность, комплаенс
- Фото не передавать лишним сторонам сверх нужды; не оверселлить AI; человек — финальная ответственность.

## 9. Аналитика
- Доля принятых AI-предложений, правки, время создания, рост supply.

## 10. Доступность и i18n
- Доступные подсказки; i18n `ai_listing.*` (NL/FR/EN); описания на локали пользователя.

## 11. Тестирование
- Unit: маппинг ответа AI→поля. e2e: фото → черновик → правка → публикация; низкая уверенность → ручной ввод.

## 12. План внедрения (Rollout)
- **LATER:** после стабильного trust-loop. MVP: категория+заголовок+описание; затем атрибуты+цена. Усилие: M.

## 13. Открытые вопросы и решения
- Вендор/модель vision. Стоимость на листинг. Хранить ли фото для дообучения (приватность).

## 14. Критерии готовности (DoD)
- [ ] Из фото генерится редактируемый черновик; без авто-публикации.
- [ ] Низкая уверенность → деградация к ручному вводу.
- [ ] Аналитика принятия; i18n.
- [ ] Тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

Ревью: критичных замечаний по этому PRD нет; применить сквозные F-требования где релевантно. Релевантные: при отправке фото во внешний vision-LLM — DPA с вендором и минимизация (часть F4 GDPR-фундамента); никогда не авто-публиковать без подтверждения человека (уже в §5 — согласуется с UX-принципом «AI всегда "Проверьте и опубликуйте"»); SEO-валидации сгенерированных title/description наследуются из PRD 31; цена берётся из PRD 52 (см. его data-требования: backoff + робастная статистика). Точку интеграции согласовать с воронкой создания PRD 31 (черновик-до-верификации).
