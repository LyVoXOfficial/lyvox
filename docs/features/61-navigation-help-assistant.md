# PRD: Помощник навигации (one-time onboarding-подсказка по разделам)

> **Статус кода:** ⛔ НЕ НАЧАТО (есть паттерны: cookieConsent store, localStorage-хелперы, i18n; свайп-coach-mark из [[01-discover-swipe-system]] — частный случай этого механизма)
> **Категория:** Бонус / UX-онбординг
> **Приоритет:** P2 (но дёшево и сильно влияет на первое впечатление)
> **Зависит от:** [[42-i18n-localization]], [[01-discover-swipe-system]] (coach-mark), [[30-identity-account]] (хранение в профиле для залогиненных)
> **Последнее обновление:** 2026-06-27

---

## 1. Зачем это (Problem & Outcome)
- **Проблема.** Новый пользователь, впервые попадая в раздел (главная, поиск, свайп-лента, создание объявления, чат, профиль), не всегда понимает, что где и как пользоваться. Особенно важно для trust-first маркетплейса с непривычными элементами (свайпы, identity-gating, safe-deal). Постоянные подсказки раздражают; полное их отсутствие — теряет новичков.
- **Исход (как просил пользователь).** При **первом** входе в раздел система **один раз** предлагает помощь по навигации: либо просто спрашивает «нужна ли помощь», либо сразу начинает мягкий тур — **с возможностью выключить**. Повторно **автоматически не предлагает никогда**. Пользователь может **вручную открыть** помощь снова в любой момент.
- **KPI:** доля принявших помощь; завершаемость тура; влияние на конверсию раздела (продажа/контакт/листинг); доля «больше не предлагать»; повторные ручные запуски.

## 2. Контекст и текущее состояние (As-is)
- Нет общего механизма онбординг-подсказок. Есть близкие паттерны: `lib/cookieConsent/store.ts` (consent), localStorage-хелперы (`seenAdverts.ts`, `recentlyViewed.ts`), i18n (`useI18n`/`t()`).
- Свайп coach-mark (PRD 01) — частный случай; этот PRD обобщает механизм на все разделы.

## 3. Пользователи и сценарии (User stories)
- **Новичок (гость/registered):** «Захожу в раздел впервые → мне один раз предлагают помощь; могу принять, отказаться или отключить навсегда».
- **Любой:** «Хочу заново открыть подсказку вручную, когда забыл, как что работает».
- **Опытный/повторный:** «Не хочу, чтобы мне это всплывало повторно — и оно не всплывает».

## 4. UX и поведение (детально)

### 4.1 Триггер и однократность
- Подсказка показывается **один раз на раздел** при первом входе. «Раздел» = логическая зона: `home`, `search`, `discover`, `post`, `chat`, `profile`, `ad` (можно расширять). Ключи раздельные, чтобы помощь была контекстной.
- После **любого** исхода (принял / отказался / отключил / просто закрыл) — флаг «offered:<section>» ставится, и **авто-показ больше не повторяется** для этого раздела.

### 4.2 Шаг 1 — мягкое предложение (non-blocking)
- Небольшая карточка/тост снизу (не модалка во весь экран, не блокирует контент): «Показать, как пользоваться этим разделом?»
- Кнопки: **«Показать»** (старт тура) · **«Не сейчас»** (скрыть, флаг offered ставится) · **«Больше не предлагать»** (глобальное отключение всех авто-подсказок).
- Появляется с небольшой задержкой/после первого взаимодействия (не мгновенно на загрузке), чтобы не мешать.

### 4.3 Шаг 2 — тур (если «Показать»)
- Пошаговые подсветки ключевых элементов раздела (coach-mark/tooltip с затемнением фона вокруг целевого элемента): для discover — что делают свайпы (ссылка на [[01-discover-swipe-system]]); для search — фильтры/сохранить поиск/свайп-режим; для post — шаги мастера; для ad — как связаться безопасно; и т.д.
- Управление: «Далее» / «Назад» / «Пропустить» (в любой момент — выключить). Прогресс «2 из 5».
- По завершении/пропуске — флаг completed/skipped; авто-тур больше не запускается.

### 4.4 Ручной повторный запуск
- В шапке/меню каждого раздела — иконка «?» (help). Клик → запускает тур по требованию (в обход флага offered). Это **единственный** способ увидеть помощь снова — автоматически она не появляется.
- В Настройках → «Помощь и подсказки»: тумблер «Предлагать подсказки в новых разделах» (вкл/выкл глобально) + кнопка «Сбросить все подсказки» (для онбординга заново, например на новом устройстве/демо).

### 4.5 Состояния
- not_offered → offered(declined/disabled) → (manual) running → completed/skipped.
- Глобальный тумблер off → шаг 1 не показывается нигде (но «?» работает).

### 4.6 Копирайт (примеры, RU; локализовать NL/FR/EN)
- Заголовок: «Нужна помощь с этим разделом?»
- Кнопки: «Показать» · «Не сейчас» · «Больше не предлагать».
- В туре: короткие подписи к элементам (1 строка каждая).

## 5. Логика и правила
- **Однократность авто-показа** на раздел — ключевое правило (никогда не авто-повторять).
- Уважать **глобальный тумблер** и **consent** (`functional`-cookie, как taste-модель) — без согласия на функциональные cookie для гостя храним только в памяти сессии или не показываем повторную защиту (решить, см. §13).
- Не показывать одновременно с cookie-баннером/другими модалками (очередь/приоритет).
- Не показывать, если пользователь сразу начал активно действовать (анти-раздражение) — показывать в «паузе».
- Краевые: смена языка во время тура; быстрый уход со страницы; SSR (показ только клиентски, без layout shift).

## 6. Данные (Data model)
- **Гость:** `localStorage` ключ `lyvox:nav-help` = `{ globalEnabled: bool, sections: { [key]: 'offered'|'completed'|'skipped'|'disabled' } }`.
- **Залогиненный (кросс-девайс):** хранить в `profiles.onboarding_prefs JSONB` (новое поле; миграция `YYYYMMDDHHMMSS_onboarding_prefs.sql`; RLS owner-only; учесть `lock_profiles_columns`). Синхронизация localStorage ↔ профиль при логине (merge: если где-то «offered/disabled» — уважать).
- Экспорт/удаление — в DSAR ([[41-gdpr-legal]]).

## 7. API / интеграции
- Для залогиненных — чтение/запись через `PATCH /api/profile/update` (+zod) или `/api/me`. Для гостя — без сети.
- Переиспользовать общий компонент тура (один движок для всех разделов и для свайп-coach-mark [[01-discover-swipe-system]]).

## 8. Безопасность, приватность, комплаенс
- Только UX-предпочтения, не чувствительные данные; consent для функционального хранения; в экспорт/удаление аккаунта.

## 9. Аналитика и события
- `navhelp_offered` {section}, `navhelp_accepted`/`navhelp_declined`/`navhelp_disabled` {section}, `navhelp_tour_step` {section, step}, `navhelp_completed`/`navhelp_skipped` {section}, `navhelp_manual_open` {section}, `navhelp_global_toggle` {value}.
- Анализ: завершаемость тура по разделам, влияние на конверсию, доля «больше не предлагать».

## 10. Доступность и i18n
- Тур доступен с клавиатуры (Tab/Esc для выхода, стрелки для шагов), фокус-менеджмент, ARIA для подсветок и кнопок; не ломать скринридер.
- Все строки через `t()`; локали NL/FR/EN (+DE/RU). Подписи коротки и переводимы.

## 11. Тестирование
- Unit: машина состояний (offered один раз; авто-показ не повторяется; manual всегда работает; глобальный тумблер; merge гость↔профиль).
- Component/e2e: первый вход в раздел → предложение → «Показать» → тур → завершение → повторный вход (НЕ показывается) → «?» (показывается) → «Больше не предлагать» → нигде не показывается; смена языка; consent off.

## 12. План внедрения (Rollout)
- **Фаза 1 (S):** общий движок тура + механизм флагов (localStorage), интеграция в discover (заменяет/обобщает coach-mark из PRD 01) + 1-2 раздела (home, post).
- **Фаза 2 (S–M):** остальные разделы (search, chat, profile, ad), кнопка «?», настройки помощи.
- **Фаза 3 (S):** кросс-девайс хранение в `profiles.onboarding_prefs` + merge при логине.
- Фиче-флаг `nav_help`. Усилие: S–M.

## 13. Открытые вопросы и решения
**Решено (по запросу пользователя):** авто-показ строго один раз на раздел; есть выключение; ручное переоткрытие; авто-повтора нет никогда.
**Нужно решить:**
1. Гранулярность «разделов» (список ключей) — стартовый набор: home, search, discover, post, chat, profile, ad.
2. Для гостя без functional-consent: не хранить и потому показать ещё раз при следующем визите, или хранить в session-memory? (Рекоменд.: уважать consent; без него — показывать максимум раз за сессию, не персистить.)
3. Стиль шага 1: тост vs небольшая карточка (Рекоменд.: ненавязчивый тост снизу).

## 14. Критерии готовности (DoD)
- [ ] При первом входе в раздел — однократное предложение помощи (Показать / Не сейчас / Больше не предлагать).
- [ ] Тур запускается, проходится по шагам, выключается в любой момент.
- [ ] Авто-показ **никогда** не повторяется; ручной запуск по «?» работает всегда.
- [ ] Глобальный тумблер + сброс подсказок в настройках.
- [ ] Хранение: localStorage (гость) + `profiles.onboarding_prefs` (залогиненный) с merge; consent учтён; DSAR-экспорт.
- [ ] a11y (клавиатура/ARIA/фокус) + i18n (NL/FR/EN) готовы.
- [ ] Аналитика §9; тесты §11 зелёные.

## Ревью-требования и sign-off (2026-06-28)

> Ближе всех к sign-off из UX-зоны (машина состояний, однократность, ручной перезапуск, consent, merge гость↔профиль). Нужны точечные правки по поведению предложений и a11y.

### UX
- **Глобальный rate-limit авто-предложений (главное).** Однократность «на раздел» недостаточна: при первом визите home → search → discover → post пользователь получит 4 предложения подряд = раздражение/слепота к подсказкам. Зафиксировать: **не более 1 авто-предложения за сессию / за N минут**; остальные разделы — тихо помечаются `offered` или предлагаются при следующих визитах. Усиливает §4.1/§5 (сейчас правило только «один раз на раздел»).
- **Различать активный дисмисс vs пассивный уход.** §4.1 сейчас: «после любого исхода → offered» — это жжёт помощь при случайном смахивании тоста. Решение: «закрыл крестиком/кнопкой» → `offered`; «проигнорировал/ушёл» → можно предложить ещё раз при следующем визите (максимум 2 раза). Смягчить §4.1.
- **Единый флаг coach-mark с PRD 01 (парный фикс).** Флаг Discover-coach = `sections.discover` в единой схеме движка 61; PRD 01 **не держит** отдельный `lyvox:discover:coach`. Синхронизировать владение, иначе риск двойного показа. В §7/§13.
- **Контент-приоритет тура — trust как анти-тревога.** В туре каждого раздела минимум одна карточка про безопасность («Почему продавец скрыт», «Как платить безопасно») — превращает онбординг в trust-инструмент, а не «где кнопки». В §4.3.
- **a11y (уточнить §10):** `prefers-reduced-motion` отключает анимации перехода между шагами; фокус-порядок подсветки «тултип → действие → закрыть»; тост шага 1 — `role=status`/`aria-live`, не крадёт фокус; подсветка целевого элемента видна при высоком контрасте (рамка + затемнение, не только цвет); тач-таргеты «Далее/Назад/Пропустить/Понятно» ≥44px; Esc выходит и возвращает фокус.

**Вердикт:** 🔄 — нужны: глобальный rate-limit (≤1/сессия), обработка пассивного дисмисса, синхронизация флага с PRD 01, trust-контент тура, пара a11y-уточнений. После правок → ✅.
