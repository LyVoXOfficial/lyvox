## 1. Зачем это (Problem & Outcome)
- **Проблема.** Универсальные поля не дают релевантного поиска и доверия. Каждой категории нужны свои атрибуты, словари, валидации и бельгийские документы.
- **Исход.** Структурированные категории с обязательными атрибутами и локальными документами (Car-Pass, EPC/PEB и т.д.), что улучшает поиск, доверие и комплаенс.
- **KPI:** полнота атрибутов листингов; использование атрибут-фильтров; качество выдачи.

## 2. Контекст и текущее состояние (As-is)
- Транспорт: `vehicle_catalog`, `vehicle_reference_tables`, каскад make/model/year/body, i18n словари (`vehicle_make/model/generation_i18n`), insights (`generation_insights`).
- Недвижимость: `real_estate_catalog` (типы, EPC/PEB ratings — `/api/catalog/epc-ratings`, property-types).
- Работа: `jobs_catalog` (contract-types, CP-codes, job-categories).
- Электроника: `electronics_extended` (device brands/models — `/api/catalog/device-*`).
- Общая каталог-схема: `catalog_schema_tables/seed`, `/api/catalog/*`, `/api/catalog/schema`.
- Стратегия скоупа: MVP-категории — транспорт, электроника, дом/детское/мода; недвижимость/работа/услуги/животные — позже.

## 3. Пользователи и сценарии
- **Продавец:** «Заполнить нужные поля для моей категории (марка/модель, EPC, тип контракта…)».
- **Покупатель:** «Фильтровать по атрибутам категории».

## 4. UX и поведение (детально)
- Каскадные селекторы (транспорт), словари (локализованные), обязательные/опциональные поля по схеме категории.
- Бельгийские документы: Car-Pass чек-лист (транспорт), EPC/PEB (недвижимость) — поля/чек-листы.
- Кэширование схемы (улучшение — сейчас re-fetch на каждую смену категории).

## 5. Логика и правила
- Схема категории определяет поля/валидации/обязательность.
- Локализация имён собственных — из curated CSV, **не** машинный перевод (см. TODO i18n).
- Краевые: отсутствующие словарные значения, новые модели/бренды.

## 6. Данные (Data model)
- Категорийные таблицы (выше) + `categories` дерево + i18n-таблицы. Индексы по атрибутам для фильтров.
- RLS: публичное чтение словарей; запись — админ/сид.

## 7. API / интеграции
- `/api/categories/tree`, `/api/catalog/{schema,fields,device-brands,device-models,contract-types,cp-codes,job-categories,property-types,epc-ratings}`.

## 8. Безопасность, приватность, комплаенс
- Car-Pass (транспорт), EPC/PEB (недвижимость) — документные чек-листы; GPSR-поля для профи ([[15-recupel-dac7-compliance]]/[[38-moderation-reports]]).

## 9. Аналитика
- Использование атрибут-фильтров; полнота заполнения по категориям.

## 10. Доступность и i18n
- Локализованные словари (NL/FR/EN/DE/RU); коды (body/fuel/transmission) → словари UI; hreflang ([[42-i18n-localization]]).

## 11. Тестирование
- Unit: схема→поля, каскады. API: словарные эндпойнты. e2e: листинг по каждой MVP-категории.

## 12. План внедрения (Rollout)
- **Поддержка:** кэш схемы; добить i18n словарей (vehicle CSV, de-rollout); активировать категории по стратегии скоупа. Усилие: S–M.

## 13. Открытые вопросы и решения
- Порядок активации категорий вне MVP. Источник curated CSV для i18n имён.

### §13 — SEO-парковка (отложено до SEO-прохода)
- `generateMetadata` на `/c/*`: per-locale title/description с BE-ключами; self-canonical; `alternates.languages` + hreflang (`nl-BE/fr-BE/en/de-DE/ru/x-default`).
- `<h1>` = локализованное имя категории (ровно одно H1 на странице).
- `CollectionPage` JSON-LD на `/c/*` (BreadcrumbList уже есть).
- Пагинация `/c/{path}?page=N` — self-canonical на каждой странице (НЕ canonical на стр.1).
- SEO-перелинковка: дочерние категории → уже есть (`<CategoryList>`), популярные города — добавить позже.
- Facet-стратегия: фасеты (`?fuel=` и т.п.) → `noindex`; среднесрочно — whitelist ценных фасетов.
- Sitemap: все активные категории с `lastmod` (зависит от P0-фундамента аудит-02).

## 14. Критерии готовности (DoD)

- [ ] Локальные документы (Car-Pass/EPC) как поля/чек-листы.
- [ ] Схема кэшируется; i18n словарей корректен.

## Ревью-требования и sign-off (2026-06-28)

### SEO (БЛОКЕР — фундамент F12)
- **Категорийные страницы `/c/*` — главный SEO-актив маркетплейса** (под «BMW te koop», «appartement à louer Gent»), но SEO в PRD отсутствует как класс. Сейчас `/c/*` наследует root `title=LyVoX` (дубли). Добавить новый под-раздел «SEO `/c/*`»:
  - **`generateMetadata` per `/c/*`:** title `{CategoryName} {BE-ключ} in België · LyVoX` per-locale; уникальный description с BE-ключами NL/FR (§0.3 SEO-ревью: `tweedehands/te koop/occasion/à vendre` + city-токены `Antwerpen/Anvers`…); self-canonical через `absoluteUrl()`; `alternates.languages` + hreflang (`nl-BE/fr-BE/en/de-DE/ru` + `x-default`).
  - **`<h1>`** = локализованное имя категории (ровно одно H1).
  - **JSON-LD:** `BreadcrumbList` (данные уже строятся в `buildCategoryBreadcrumbs`) + `CollectionPage`/`ItemList` первых N объявлений с `position`.
  - **Пагинация:** `?page=N` индексируемы с self-canonical (НЕ canonical на стр.1).
  - **SEO-перелинковка:** хлебные крошки, дочерние категории, «популярные подкатегории/города» (источник — `/api/categories/tree`).
  - **Facet-стратегия:** на старте все фасеты (`?fuel=`, `?brand=`) → `noindex`, индексируем только `/c/{path}`; среднесрочно — whitelist ценных фасетов как ЧПУ-лендингов с уникальными метаданными.
  - **Вклад в sitemap:** все активные категории (`is_active`) с `lastmod`.
- **Зависимость:** до закрытия P0-фундамента из аудита 02 (metadataBase, robots/sitemap, og:image, canonical-фикс) DoD не считается выполненным.
- **DATA-замечание (фундамент F13):** `catalog_groups` (конфиг раскладки tab/section) и `catalog_fields` — источник полей для structured data в [[62-product-category-detail]]; добавить таблицу `catalog_groups(domain, group_key, display, tab_key, tab_order, ...)`.

**Вердикт:** ⛔ Блокер по SEO — без per-category метаданных/H1/BreadcrumbList/ItemList/перелинковки органическая стратегия не работает. После добавления под-раздела SEO `/c/*` → 🔄/✅.
