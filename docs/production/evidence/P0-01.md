## Исходный дефект

- Последние 20 запусков CI падали до checkout: workflow ссылался на несуществующие commit SHA `actions/checkout` и `actions/setup-node`.
- `gitleaks/gitleaks-action@v2.3.4` был mutable и требовал отдельную лицензию для organization repository.
- `main` не имел branch protection или ruleset.
- Commit `d451af85fbf3639a72ef9a1b2c5b8a5318cf9951` получил красный GitHub CI, но затем успешный Vercel Production deployment. Следовательно, прежний deploy не зависел от CI.

## Реализовано

- Все внешние Actions закреплены существующими immutable commit SHA; repository policy дополнительно требует SHA pinning.
- Permissions GitHub Actions ограничены read-only, разрешены только GitHub-owned Actions и `pnpm/action-setup`.
- Dependency audit читает JSON для production и полного workspace, блокирует `critical`, `high`, любой advisory пакета `next`, schema drift, muted advisory и скрытую audit-suppression конфигурацию.
- Gitleaks v8.30.1 скачивается из official release и проверяется SHA256 до запуска полного history scan.
- `CI Success` зависит от lint/typecheck, build/bundle budget, i18n, tests, migration/security guards, audit и secret scan.
- Активирован repository ruleset `main-production-gate` (`id: 18791401`): только PR, strict `CI Success` от GitHub Actions app `15368`, обязательный `Preview`, squash-only, запрет delete/force-push.
- В GitHub включены Dependabot security updates, secret scanning и push protection.
- Production environment допускает deployment только из `main` (`branch policy id: 54335876`).
- Удалён конфликтующий root `vercel.json`; канонический Root Directory — `apps/web`.
- `apps/web/vercel.json` отключает автоматический Git deploy ветки `main`.
- Новый Production workflow не имеет arbitrary-SHA/manual dispatch: он запускается только после зелёного push-run канонического workflow `id: 203931925`, `path: .github/workflows/ci.yml` на `main`; проверяет producer app `id: 15368`, exact SHA и текущий `main`.
- Deploy job не устанавливает Vercel CLI или npm dependencies и передаёт Vercel secrets только shell-шагам, которым они нужны. Встроенный Node обращается к официальному Vercel REST API.
- До мутации проверяются project/team id, Git link и public repo id `1061423103`, branch `main`, Root Directory `apps/web`, Framework `nextjs`, Node `24.x`, отсутствие custom Build/Install/Output overrides, отключённый Ignored Build Step и `autoAssignCustomDomains=false`.
- Exact SHA собирается как staged Production deployment без алиасов; каждый poll повторно валидирует project/source/SHA/CI metadata, `buildSkipped=false` и конечный `readySubstate=STAGED`.
- Непосредственно перед promotion повторяются CI/main-проверка и Vercel preflight. Изменение Production target или активный emergency rollback/promote оставляют build staged и прерывают release. Успех принимается только когда `targets.production.id` указывает на deployment и он имеет `readySubstate=PROMOTED`.

## Открытые stop-ship пункты

1. GitHub secret scanning подтверждает два открытых Google API Key alerts. Актуальное дерево очищено, но ключи присутствуют в истории и должны быть отозваны владельцем Google Cloud. После ротации нужен отдельный подтверждённый выбор: согласованная history rewrite либо fingerprint-scoped `.gitleaksignore` только вместе с rotation evidence и закрытыми GitHub alerts. До этого full-history CI останется красным; suppression заранее запрещён.
2. Добавить в GitHub environment `Production`: `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`.
3. В Vercel подтвердить canonical Git/build settings (включая отсутствие custom Build/Install/Output overrides) и выключить Settings → Environments → Production → Branch Tracking → Auto-assign Custom Production Domains. Без этого staged→promote preflight намеренно падает.
4. После ротации ключей получить общий зелёный `CI Success`, затем провести Production staged→promote proof на точном SHA и сохранить deployment URL.
5. Добавить второго независимого GitHub reviewer; затем изменить ruleset на `required_approving_review_count: 1` и `require_last_push_approval: true`.
6. Описать и проверить emergency hotfix procedure без прямого push в `main`.

## Rollback

- Код deploy workflow можно откатить обычным PR, но автоматический deploy `main` нельзя включать, пока нет эквивалентного required Deployment Check.
- Ruleset и environment policy экспортированы через GitHub API; их ослабление считается release-risk изменением и требует отдельного evidence.
