# T08 — fast_goods: /post за ≤4 шага для простых товаров

**Модель:** ТОЛЬКО сильная (gpt xhigh / opus/fable-класс) + ревью диффа отдельным заходом. Самый конверсионно-критичный флоу supply; PostForm.tsx ~1400 строк.
**Ветка:** `feat/post-fast-goods` · **Приоритет:** P1 · **Оценка:** 1-2 дня. **Если не уверен — раздели на подшаги и коммить по одному.**

## Зачем
`PostForm.tsx:58` — `TOTAL_STEPS = 8`. Для дивана/куртки это убийца supply (краш-тест агент C; SITE_BLUEPRINT волна 2). Цель: простые товары публикуются за 4 обязательных шага, тяжёлые вертикали (авто/недвижимость/работа) сохраняют полный флоу.

## Дизайн (согласован тремя документами — blueprint, бриф §11.3, репликация)
```ts
type PostFlowMode = "fast_goods" | "vehicle_deep" | "real_estate_deep" | "job_deep" | "generic";
```
- Режим выводится из выбранной категории через `detectCategoryType` (`lib/utils/categoryDetector.ts`): vehicle→vehicle_deep, real_estate→real_estate_deep, jobs→job_deep, остальные → fast_goods.
- fast_goods шаги: (1) фото [компрессия уже есть — compressImage.ts]; (2) категория+заголовок; (3) цена+состояние; (4) локация+предпросмотр→публикация.
- Качественные поля категории (схемные) для fast_goods НЕ выпадают из продукта: после публикации на странице объявления владельцу показывается баннер «Дополните объявление — его увидят больше людей» → редактирование (progressive profiling; НЕ попап-наггинг, один баннер).

## Шаги (строго по одному коммиту на пункт)
1. **Разведка**: прочитай PostForm.tsx целиком; выпиши в отчёт-комментарий ветки: какие state-поля к какому шагу привязаны, где валидация шагов, где draft-создание. НЕ правь ничего в этом коммите (коммит = только файл заметок в docs/todo/notes/T08-recon.md).
2. Введи `PostFlowMode` + функцию `resolvePostFlowMode(categoryPath)` + юнит-тесты (куртка→fast_goods, авто→vehicle_deep...). Форма ещё не тронута.
3. Ремапни шаги: массив конфигурации шагов per-mode (id шага → компонент секции), `TOTAL_STEPS` становится `steps.length`. Существующие 8 шагов = конфиг deep-режимов; fast_goods = 4. Прогресс-бар/кнопки — от длины массива. НЕ удаляй ни одну секцию — только группировка.
4. Progressive profiling баннер владельцу на /ad (компонент по образцу `PublishedShareBanner.tsx`, показывается владельцу если у объявления < N схемных полей заполнено; данные о заполненности уже приходят в page.tsx — specifics).
5. i18n: новые ключи (`post.fast_*`, баннер) в 5 локалей.
6. Финальная проверка-прогон: локально пройди ОБА флоу руками через preview (dev-сервер): куртка за 4 шага до предпросмотра; авто — полный флоу без регрессий.
