> **Ключевое отличие от старой версии:** «готово» = **проверено живьём на www.lyvox.be**, а НЕ «закоммичено».
> Каждый шаг заканчивается тем, что и код, и БД **уже на сайте**.

Выполни шаг **N** из `docs/MASTER_PRODUCTION_TZ.md` на проекте LyVoX
(Next.js 16 + React 19 + TS + Supabase + Stripe + Upstash, pnpm-монорепо, Vercel)

## 0. Загрузи контекст (до любого действия)

1. `docs/MASTER_PRODUCTION_TZ.md` → раздел «✅ TODO» → найди шаг N: название, входящие ID
   (SEC-…/PERF-…/FLAG-…) и тег 🎚️ [Модель · мощность].
2. Для КАЖДОГО входящего ID открой его детальный блок (Проблема → Решение → Критерии
   приёмки → Зависимости/Ключи). Несколько ID — прорабатывай все.
3. Прочитай `CLAUDE.md` (конвенции — ОБЯЗАТЕЛЬНЫ) и раздел «📏 Правила + журнал ошибок»
   (RULE-01…04 нарушать нельзя).
4. Есть `graphify-out/` — используй (`wiki/index.md`, `GRAPH_REPORT.md`) до широкого поиска.
5. **Свери реальное состояние ПРОДА с локальным ДО работы** (иначе «done ≠ applied»):
   - `git log --oneline main..HEAD` — что закоммичено, но не в `main`.
   - `git status -sb` — есть ли upstream у ветки (не запушена = не на сайте).
   - Если трогаешь схему: `supabase migration list --db-url "$SUPABASE_DB_URL"` — пустая колонка Remote = миграция НЕ в проде; версия без локального файла = дрейф, чинить (см. §5).

## 1. Тип задачи

- `[— · n/a внешнее]` → это НЕ код (юр/налог/ключи founder-а): подготовь бриф/чек-лист,
  отчитайся, что должен сделать основатель. Деплой не нужен. Стоп.
- `ultracode` → высший риск (деньги/безопасность/юр): работай максимально глубоко —
  широкое исследование, тесты, adversarial-самопроверка перед финалом.
- За незакрытым гейтом (money-flow за PROD-F3 / нужен API-ключ) → код за AND-gate фиче-флагом,
  прод-денежный-флоу НЕ включай; отметь блокер, остановись на границе (деплой кода-за-флагом — ок).

## 2. Золотые правила (иначе работа не принимается)

- ⛔ Никакого прод-кода денежного потока до закрытия F3 (схема/провайдер-абстракция — можно).
- Ключи — только в env; тумблер — в БД; capability ON = (тумблер === true) && (ключи присутствуют).
- RLS даёт ROW, не COLUMN (RULE-01): user-editable таблицы с trust/entitlement/tax-колонками →
  column-scoped GRANT. SECURITY DEFINER с uuid-аргументом → revoke execute from public/authenticated/anon (RULE-02).
- Три Supabase-клиента не смешивать (supabaseServer / supabaseService / browser). Service-role — только сервер.
- API-конверт `{ok,data}` (createSuccessResponse/createErrorResponse); клиент читает `body.data.X`.
- i18n: никаких хардкод-строк, всё через `t()`; 5 локалей в паритете (guard-тест).
- Миграции idempotent (`IF NOT EXISTS`/`ON CONFLICT`) + RLS; после изменения схемы — `pnpm gen:types`.
- Верифицируй на ПРОДЕ, не на local dev (RULE-04: dev CSS кешируется и врёт).

## 3. Процесс

1. Сначала ПЛАН (что меняешь, файлы, риски) — 5–8 строк.
2. TDD где применимо: тест(ы) на критерии приёмки → потом имплементация.
3. Ветка `feat/<id>-<slug>`, атомарные коммиты. Делай ТОЛЬКО этот шаг; смежное — записывай отдельно.

## 4. Верификация КОДА (перед деплоем — обязательно)

- `pnpm typecheck && pnpm test && pnpm lint` — зелёное.
- **`pnpm audit --prod` — в выводе НЕ должно быть `next` (RULE-05).** Vercel молча блокирует
  деплой на уязвимой версии Next.js. Есть — подними `next` + `eslint-config-next` до пропатченной,
  переустанови, перепроверь.
- Чистая сборка от закоммиченного дерева: `git status` чистый → `rm -rf apps/web/.next && pnpm build`
  (exit 0). `tsc`/кэш-сборка на грязном дереве не доказательство.
- Приведи вывод как доказательство. Без вывода проверок работу завершённой не выдавать.

## 5. БАЗА → ПРОД (если менялась схема — обязательно ДО деплоя кода)

Порядок: **БД раньше кода** (новый код часто ждёт новую схему).
1. `supabase db push --db-url "$SUPABASE_DB_URL" --dry-run` — увидеть pending.
2. **Дрейф** (remote-версия без локального файла / «migrations to be inserted before…»):
   - remote-only версия → создать локальный файл `<version>_*.sql`, воспроизводящий её statements
     (взять из `supabase_migrations.schema_migrations.statements` через psql), idempotent. НЕ `migration repair --reverted` (пометит живое как откатанное), НЕ `db pull` (тащит весь diff).
   - pending-файлы старше уже-применённой версии → пуш с `--include-all`.
3. `supabase db push --db-url "$SUPABASE_DB_URL" [--include-all]` — применить.
4. `supabase migration list --db-url "$SUPABASE_DB_URL"` → Local == Remote по всем строкам.
5. `pnpm gen:types`; если `supabase/types/database.types.ts` изменился — закоммить.
   ⚠️ session-pooler флаки на длинных сессиях — короткие запросы, при блокировке ищи свои `idle in transaction`.

## 6. ДЕПЛОЙ КОДА → ПРОД (это и есть «готово»)

1. Влить работу в `main` (мерж ветки в main или PR→merge) и **`git push origin main`** —
   Vercel авто-деплоит www.lyvox.be. (Дерево чистое, все проверки §4 зелёные, `next` пропатчен.)
2. Дождись готовности: `vercel ls lyvox-frontend` → `Ready` (или проверь дашборд).
3. **Верифицируй ЖИВЬЁМ на проде** (RULE-04, env boot-validation → «Ready» ≠ «отдаёт»):
   - `curl -sS -o /dev/null -w "%{http_code}" https://www.lyvox.be/` → **200** (сайт бутается).
   - **Проверяй ИМЕННО ту поверхность, которую менял этот шаг, а не только `/`.** Homepage
     не трогает auth/API/мутации — `GET /` зелёный ничего не доказывает про SEC-CSRF/BOT/VALID/RL.
     Backend/auth-шаг → curl-ни изменённый роут: страницы (`/login`, `/sell`, `/post`) → 200/307,
     мутирующие API на GET → 405/401/400 (**не 500** = роут бутается, гард не сломал загрузку).
   - `curl -s https://www.lyvox.be/<путь>` и grep маркер изменения (новый класс/строка/i18n
     с `-H "Cookie: locale=ru"`; для header-изменений — `curl -D -`). Auth-страницы редиректят на /login.
